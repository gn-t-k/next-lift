# データベースルール

DB スキーマ・ERD・マイグレーションを扱う際のルール。

## DB enum / CHECK 制約は使わない

- DB の enum 型や CHECK 制約で値セットを制限しない
- 区分値（`category` / `status` / `type` 等）はアプリ側の定数（TypeScript の union 型等）で管理する

**Why:** DB 制約は値の追加にマイグレーションが必要で保守コストが高い。アプリ側のバリデーションは結局必要なので二重管理になる。すべての書き込みがアプリ経由なら、アプリ側バリデーションで十分。

**How to apply:** ERD 設計やスキーマ定義で区分値を扱う際、DB の enum/CHECK ではなくプレーンな varchar として定義し、値の制御はアプリ側に委ねる。

## JOIN の複雑さは正規化を避ける理由にならない

- 「JOIN が増えてクエリが複雑になる」を正規化やテーブル分割を避ける理由として挙げない
- クエリが書けないレベルで破綻するときだけ考慮する

**Why:** AI エージェントと協働で開発する場合、SQL とテストは一度書けば済む。適切な index を設定すればパフォーマンスも問題にならない。

**How to apply:** DB 設計の判断で、テーブル分割のデメリットとして「JOIN が複雑になる」を出さない。
