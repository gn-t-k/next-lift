# コミット・ブランチ分割戦略

## ブランチ分割の手順

1. 各変更の目的を1行で書き出す（未コミット diff・既存コミットの両方が対象。1ファイルに複数の目的があれば分ける）
2. 同じ目的の変更をグルーピングする
3. グループごとに1ブランチとする

判断基準（PRの diff が最小限になるように）:

- 変更の目的（branch-naming の type）が異なる → 別ブランチ。1 ブランチ = 1 type を守り、既存ブランチに別 type が紛れていないかも見る
- 影響範囲が独立している → 別ブランチ
- 個別にレビューすべき内容 → 別ブランチ
- 迷ったら分割する（まとめる判断の方がハードルが高い）

例:

- 振る舞いの変更 + 同範囲のリファクタ → 別々のブランチ
  - リファクタ側のレビューを「移動先で振る舞いを保つ」確認に集中させ、振る舞い変更側のレビューをユーザー影響・デザイン意図に集中させるため
- 新機能 + 開発ツールの改善 → 別々のブランチ
- 依存関係の更新 + 設定ファイルの追加 → 別々のブランチ

## ベースブランチの決定

ブランチを分割した後、各ブランチのベースブランチを決定する:

1. ブランチ間に依存関係がない → base は `main`
2. ブランチBがブランチAの変更を前提としている → base はブランチA

依存関係の例:

- ブランチAでファイルを追加し、ブランチBでそのファイルを変更・削除する
- ブランチAで導入した機能を、ブランチBが利用している
- 振る舞いを変更するブランチを土台に、その周辺をリファクタするブランチを積む

## コミット分割の基準

レビューしやすい粒度で分割:

- 機能的に関連性のある変更をまとめる
- 無関係な変更を同じコミットに含めない
- 細分化しすぎて変更の目的がわかりにくくならないように注意

## 既存コミットの履歴整理（rebase）

現ブランチに既にいくつかコミットが積まれているとき、未 push の範囲なら整理を提案する余地がある:

- WIP / fixup 系のコミットを squash する
- 順序を入れ替えて論理的に読みやすくする
- 後段のコミットで打ち消されている変更を統合する
- 別 type が紛れたコミットは別ブランチへ切り出す（新ブランチに cherry-pick → 元ブランチから drop して force-with-lease）

### 安全な対象範囲

整理してよいのは **未 push のコミットだけ**:

```bash
git log @{upstream}..HEAD --oneline
```

このリストの範囲内であれば、ローカルでの履歴書き換えは安全（誰のリモート作業も壊さない）。

### 触ってはいけないケース

- `main` ブランチ上での rebase（共有ブランチの履歴を壊す）
- 既に push 済みでレビュー中の PR ブランチ（force-with-lease の影響をユーザーが理解していない限り提案しない）
- 他人がベースにしているブランチ（依存ブランチがある場合）

これらに該当するときは rebase を提案せず、新規 fixup コミットなどで対応する。

## 返却フォーマット

以下の形式で提案を返す:

```markdown
## 提案

### ブランチ: `<type>/<description>` (base: `main`)

#### コミット1: <日本語メッセージ>
- `path/to/file1.ts`: <変更概要>
- `path/to/file2.ts`: <変更概要>

#### コミット2: <日本語メッセージ>
- `path/to/file3.ts`: <変更概要>

---

### ブランチ: `<type>/<description>` (base: `<依存先ブランチ or main>`)

...

### 履歴整理（任意）

未 push の範囲で以下の rebase を推奨:
- `<sha> <message>` を `<別の sha>` に fixup
- ...
```

差分の詳細コードではなく、**構造化された提案**を返す。ファイルパスと変更概要のみを含める。コミットメッセージは日本語で「なぜ、何を変更したか」を明確に。曖昧な点があれば質問として返す。
