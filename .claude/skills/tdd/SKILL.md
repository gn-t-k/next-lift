# TDD - テスト駆動開発スキル

Kent BeckのTDDサイクルでタスクを1つずつ実装する。各ステップでユーザーレビューを挟み、進捗をファイルに記録する。

## 原則

- **Red→Green→Refactor**: テストを書く→通す→きれいにする。このサイクルを崩さない。テストなしの実装はフィードバックループを失い、意図しない挙動を見逃す
- **要所でレビュー**: テストの方向性はユーザーが、実装の品質はサブエージェントが、最終形はユーザーが確認する。ユーザーの判断が必要なのは「何をテストするか」と「最終的なコードの形」であり、途中の最小実装ではない
- **都度記録**: タスクの進捗と判断をファイルに書き込む。コンテキストクリアで進捗が失われると、同じ作業を繰り返すことになるため

## 開始時

### ToDoリストの準備

呼び出し元からToDoリストの準備方法が指定されている場合は、その指示に従う。

指定がない場合:

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. `references/todo-list-template.md` に基づいてToDoリストファイルを作成する
3. 実装対象のタスク一覧をユーザーに確認し、ToDoリストに記入する

## TDDサイクル

### タスクの選択

1. ToDoリストから次の `pending` タスクを選ぶ
2. ステータスを `in_progress` に更新する

### Red: テストを書く

1. タスクの完了条件に基づいてテストを書く
2. テストが失敗することを確認する（`pnpm test` で対象ファイルを実行）
3. **ユーザーにレビューを求める**
   - テストの意図と内容を簡潔に説明する
   - 承認を得てから次へ進む
   - フィードバックがあれば修正する

### Green: テストを通す

1. テストが通る最小限の実装を書く
2. テストが通ることを確認する

### Refactor: きれいにする

1. Exploreエージェント（subagent_type: Explore）で実装をレビューする
   - 観点: 可読性、重複、命名、既存コードとの一貫性
2. レビュー指摘がなければ、その旨をユーザーに伝えてスキップする
3. レビュー指摘があれば、指摘に基づいてリファクタリングを実施する
4. テストが通ることを確認する
5. **リファクタリング結果をユーザーにレビューさせる**
   - 何をどう変えたか、なぜ変えたかを簡潔に説明する
   - フィードバックがあれば修正する

### タスクの完了

1. `pnpm type-check && pnpm lint && pnpm test` で検証する
2. ToDoリストのステータスを `done` に更新する
3. 要点をメモに記録する（記録基準は後述）
4. 次のタスクへ進む

## 作業メモの記録基準

タスク完了時に1〜2行で記録する:

- 計画と異なる判断をした場合: 選択した方針とその理由
- 設計判断が生まれた場合: 判断の内容と理由（テスト設計、リファクタリング方針、モック戦略など）
- 問題が発生した場合: 問題の内容と対処
- 計画通りの場合: 「計画通り完了」のみ

## 全タスク完了時

1. `pnpm type-check && pnpm lint && pnpm test` が通ることを最終確認する
2. ToDoリストの全タスクが `done` であることを確認する
3. ユーザーに完了を報告する
