# Dev Workflow

フェーズ分けされた開発ワークフロー。各フェーズでユーザー承認を得てから次へ進む。

## フェーズ概要

```
要件定義 → 設計・タスク分解（/breakdown） → 実装（TDD） → レビュー
```

## 開始時

1. `docs/development/[yyyy-mm-dd-feature-name]/` ディレクトリを作成
2. 以降、各フェーズの成果物をこのディレクトリに保存

## コンテキスト引き継ぎの原則

各フェーズ間でコンテキストクリアされる前提で、成果物ファイルだけで次のフェーズを開始できるようにする。

成果物に何を書くかは「次のエージェントのコンテキストウィンドウの節約」と「情報の正確性」のトレードオフで判断する。

**書くべき情報:**
- ユーザーとの議論で合意した結論・判断（再現不可能）
- 関連するGitHub Issue番号など、情報源への参照（あれば）
- 多くの調査を経て得た結論（次のエージェントの調査コストを削減）

**書かなくてよい情報:**
- 1回の検索やコマンドで得られる事実（バージョン番号、APIシグネチャ等）
- コードベースを読めばわかるファイル一覧や現在の実装詳細（陳腐化リスクが高い）

## フェーズ1: 要件定義

`requirements` スキルを使用する。出力先は本ワークフローのディレクトリ、ファイル名は `01-requirements.md` を指定する。

### 完了条件

- [ ] 要件をユーザーが承認（`01-requirements.md` が完成）

## フェーズ2: 設計・タスク分解

### 目的

変更の方向性を決め、TDDで着手可能なタスクに分解する。

### 手順

`/breakdown` スキルを呼び出す。呼び出し時に以下を伝える:

- 要件ドキュメントのパス: `[開発ディレクトリ]/01-requirements.md`
- 出力先ディレクトリ: `[開発ディレクトリ]`
- 出力ファイル名: `02-breakdown.md`

### 完了条件

- [ ] breakdownスキルのステータスが「確定」になっている

## フェーズ3: 実装

### 目的

TDDサイクルでタスクを実装する。

### 手順

`/tdd` スキルを呼び出す。呼び出し時に以下を伝える:

- ToDoリストの準備方法: `02-breakdown.md` のタスクツリーからタスク一覧をコピーし、`[開発ディレクトリ]/03-implementation-log.md` に新規作成する
- 各タスクの説明・完了条件は `02-breakdown.md` の内容をそのまま使う

### 完了条件

- [ ] 全タスク完了（03-implementation-log.md の全タスクが `done`）
- [ ] `pnpm type-check && pnpm lint && pnpm test` が通る

## フェーズ4: レビュー

### 目的

実装完了後、品質を確認する。

### 手順

`multi-review`スキルでレビューをするかどうかユーザーに質問する。`multi-review`スキルでレビューする必要がなければ、Exploreエージェント（subagent_type=Explore）でレビューを行う。

### 確認観点

- 既存の仕様との整合性
- 既存の実装との一貫性
- テストの網羅性（カバレッジ80%目安）
- 一般的なコードレビュー（保守性・パフォーマンス・セキュリティなど）

### 完了条件

- [ ] レビュー指摘事項を解消
