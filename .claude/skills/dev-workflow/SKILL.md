# Dev Workflow

フェーズ分けされた開発ワークフロー。各フェーズでユーザー承認を得てから次へ進む。

## フェーズ概要

```
要件定義 → 設計 → 実装計画 → 実装（TDD） → レビュー
```

## 開始時

1. `docs/development/[yyyy-mm-dd-feature-name]/` ディレクトリを作成
2. 以降、各フェーズの成果物をこのディレクトリに保存

## コンテキスト引き継ぎの原則

各フェーズ間でコンテキストクリアされる前提で、成果物ファイルだけで次のフェーズを開始できるようにする。

成果物に何を書くかは「次のエージェントのコンテキストウィンドウの節約」と「情報の正確性」のトレードオフで判断する。

**書くべき情報:**
- ユーザーとの議論で合意した結論・判断（再現不可能）
- 関連するGitHub Issue番号など、情報源への参照（あれば）
- 多くの調査を経て得た結論（次のエージェントの調査コストを削減）

**書かなくてよい情報:**
- 1回の検索やコマンドで得られる事実（バージョン番号、APIシグネチャ等）
- コードベースを読めばわかるファイル一覧や現在の実装詳細（陳腐化リスクが高い）

## フェーズ1: 要件定義

`requirements` スキルを使用する。出力先は本ワークフローのディレクトリ、ファイル名は `01-requirements.md` を指定する。

### 完了条件

- [ ] 要件をユーザーが承認（`01-requirements.md` が完成）

## フェーズ2: 設計

### 目的

変更の方向性を決め、ユーザーと合意する。詳細設計はTDDで創発的に決定する。

### 手順

1. 影響範囲を特定（apps/, packages/, DB, 外部API）
2. アプローチを決定（既存拡張 or 新規追加）
3. 技術的意思決定が必要なら ADR を作成
4. `02-design.md` に保存

### テンプレート

`references/design-template.md` を使用。

### 完了条件

- [ ] 設計をユーザーが承認

## フェーズ3: 実装計画

### 目的

実装をタスクに分割し、各タスクの検証方法を決める。

### 手順

Planエージェント（subagent_type=Plan）を使用して計画を立案。

計画に含める内容：

- 必要なモジュール/変更箇所の特定
- スモールステップへの分解
- 各ステップの検証方法（フィードバックループ）
- 依存関係と実装順序

### 検証方法の選択基準（優先順位順）

#### 優先度1: 型チェック

型システムで検証可能な場合は型チェックのみで進める。

例：
- 関数のシグネチャ変更
- 新しい型の追加
- インターフェースの定義

#### 優先度2: テスト

型だけでは検証できない振る舞いがある場合はテストを書く。

例：
- ビジネスロジックの正確性
- エッジケースの処理
- 複数の関数の連携

#### 優先度3: 手動確認

テストでも検証が難しい場合は手動で確認する。

例：
- UIの見た目・レイアウト
- ブラウザ固有の動作
- 外部サービスとの連携

手動確認が必要な場合：
- Chrome DevTools MCPでブラウザを操作
- またはユーザーに確認を依頼

### テンプレート

`references/implementation-plan-template.md` を使用。

### 成果物

`03-implementation-plan.md` に保存。

### 完了条件

- [ ] 実装計画をユーザーが承認

## フェーズ4: 実装

### 目的

TDDサイクルでタスクを実装する。進捗をファイルに記録し、コンテキストクリア後もシームレスに再開できるようにする。

### 開始時

1. `04-implementation-log.md` を作成（テンプレート使用）
2. `03-implementation-plan.md` のタスク一覧を転記し、全タスクを `pending` ステータスで記録

### 手順

各タスクについて：

1. `04-implementation-log.md` のステータスを `in_progress` に更新
2. テスト・型定義を先に書く
3. ユーザーにレビューさせ、合意を得る
4. テストが通る最小限の実装を書く
5. リファクタリング（テストが通る状態を維持）
6. 検証方法に従って確認
7. `04-implementation-log.md` のステータスを `done` に更新し、要点を記録

### 作業ログの記録内容

タスク完了時に1〜2行で以下を記録する:

- 実装中に生じた判断事項（計画と異なる点、選択した方針とその理由）
- 発生した問題と対処（エラー、想定外の挙動など）
- 記録不要: 計画通りに完了した場合は「計画通り完了」のみでよい

### コンテキストクリア後の再開

コンテキストがクリアされた場合:

1. `04-implementation-log.md` を読み、現在のステータスを把握
2. `in_progress` のタスクがあれば、そのタスクの要点を確認して続行
3. 次の `pending` タスクから再開

### テンプレート

`references/implementation-log-template.md` を使用。

### 完了条件

- [ ] 全タスク完了（04-implementation-log.md の全タスクが `done`）
- [ ] `pnpm type-check && pnpm lint && pnpm test` が通る

## フェーズ5: レビュー

### 目的

実装完了後、品質を確認する。

### 手順

`multi-review`スキルでレビューをするかどうかユーザーに質問する。`multi-review`スキルでレビューする必要がなければ、Exploreエージェント（subagent_type=Explore）でレビューを行う。

### 確認観点

- 既存の仕様との整合性
- 既存の実装との一貫性
- テストの網羅性（カバレッジ80%目安）
- 一般的なコードレビュー（保守性・パフォーマンス・セキュリティなど）

### 完了条件

- [ ] レビュー指摘事項を解消
