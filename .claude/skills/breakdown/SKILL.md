# Breakdown - 要件タスク分解スキル

要件を幅優先で段階的に具体化し、TDDで着手可能なタスクのツリー構造チェックリストを生成する。

## 原則

- **幅優先で具体化**: 1つのタスクを深掘りする前に、同じ階層のタスクを一通り洗い出す。全体像を維持したまま具体化を進めるため
- **TDDが始められる粒度**: 各タスクは「テストが書ける」程度に具体的であればよい。詳細設計はTDDで創発的に決定するため、実装方法の詳細までは踏み込まない
- **検証は自動化優先**: 型システム > テスト > 手動確認の優先順で検証方法を選ぶ（詳細は `references/verification-guide.md`）。フィードバックループは短いほうがよい
- **都度記録**: 具体化の進捗をファイルに書き込み、コンテキストクリアに備える。どこまで具体化が進んでいるか見失わない

## 開始時

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力ファイル名はデフォルトで `breakdown.md` とする
3. 呼び出し元から出力先ディレクトリやファイル名が指定された場合は、その指定に従う
4. 出力先に `references/output-template.md` に基づいてファイルを作成する

## フェーズ1: 要件の確認とコードベース調査

### 目的

分解対象の要件を正確に把握し、関連するコードベースの現状を理解する。

### 手順

1. 要件ドキュメント（`requirements.md` 等）があれば読み込む
2. コードベースを調査する（Agentツール subagent_type: Explore）
   - 関連する既存実装、影響範囲、技術的制約を把握する
   - 理由: 要件とコードの両方を把握した上で質問・確認を行うため
3. 要件の全体像を把握し、不明点があればAskUserQuestionで確認する（1ラウンド2〜3問まで）
4. 分解のスコープ（どこまでを対象とするか）をユーザーと合意する
5. `breakdown.md` の「要件サマリー」セクションを更新する

### 完了条件

- 分解対象の要件が明確になっている
- スコープが合意されている
- 関連するコードベースの現状を把握している

## フェーズ2: 設計判断

### 開始条件

- フェーズ1の完了条件を満たしている

### 目的

変更の方向性を決め、ユーザーと合意する。

### 手順

1. フェーズ1の調査結果をもとに、判断が必要な設計論点を洗い出す
   - 影響範囲を特定（apps/, packages/, DB, 外部API）
   - 関連ADRがあれば確認（`docs/architecture-decision-record/`）
2. 各論点について、AskUserQuestionで選択肢を提示し、ユーザーに判断を委ねる
   - 各選択肢のpros/consはdescriptionに記載する
   - 推薦する選択肢がある場合はlabelに「(Recommended)」を付けて先頭に配置する
   - エージェントが設計を決定しない。選択肢の整理と推薦理由の提示までが役割
3. ユーザーの判断を受けて、設計方針を確定する
   - 技術的意思決定が必要ならADR作成を提案
4. `breakdown.md` の「設計概要」セクションを更新する

### 完了条件

- 影響範囲とアプローチが明記されている
- ユーザーが設計を承認している

## フェーズ3: 第1階層の分解

### 開始条件

- フェーズ2の完了条件を満たしている

### 目的

要件を大きな機能単位に分解する。

### 手順

1. 設計で決まったアプローチに基づき、要件を主要な機能単位に分解する
2. 各機能単位の依存関係と推奨実装順序を特定する
3. `breakdown.md` の「タスクツリー」セクションに第1階層を書き込む

### 完了条件

- 要件が3〜7個の機能単位に分解されている
- 依存関係と実装順序が明記されている

## フェーズ4: 幅優先の具体化

### 開始条件

- フェーズ3の完了条件を満たしている

### 目的

タスクツリーの未展開ノードを幅優先で具体化し、TDDで着手可能な葉タスクにする。

### 手順

1. タスクツリーから未展開（`[?]`マーク）のノードを上から順に1つ選ぶ
2. 必要に応じてコードベースを調査する
3. そのノードをサブタスクに分解する（具体化ルールの詳細は `references/tree-notation.md` を参照）
   - サブタスクがTDDで着手可能な粒度なら葉タスク（`[ ]`マーク）にする
   - まだ大きければ未展開ノード（`[?]`マーク）にして次のラウンドで分解する
4. 葉タスクには以下を記述する:
   - **説明**: 何をするか（1〜2文）
   - **完了条件**: 検証方法の選択基準は `references/verification-guide.md` を参照
5. `breakdown.md` のタスクツリーを更新する
6. 未展開ノードがなくなるまで繰り返す

### 葉タスクの粒度基準

- テストファイルに `describe` と `test` を書き始められる程度の具体性がある
- 1タスクの実装が概ね5〜30分で完了する規模
- 実装方法の詳細（どの関数をどう書くか等）は含めない — TDDで決める

### 完了条件

- すべてのノードが葉タスク（`[ ]`マーク）になっている
- 各葉タスクに説明と完了条件が記述されている

## フェーズ5: レビューと確定

### 開始条件

- フェーズ4の完了条件を満たしている

### 目的

タスクツリーの完全性を確認し、ユーザーの合意を得る。

### 手順

1. タスクツリー全体をレビューする
   - 要件のすべての側面がカバーされているか
   - タスク間の依存関係に矛盾がないか
   - 粒度が適切か（大きすぎ・小さすぎがないか）
2. 不足があればフェーズ4に戻って追加する
3. ユーザーに最終確認を求める
4. 合意が得られたら `breakdown.md` の「ステータス」を「確定」に更新する

### 完了条件

- タスクツリーが要件を網羅している
- ユーザーが承認している

## ラウンド終了時のステータスサマリー

各ラウンドの終了時に以下をユーザーに表示する:

```
**ステータス**
- 現在のフェーズ: X/5（フェーズ名）
- 葉タスク: N件（確定済み）
- 未展開ノード: N件（次に具体化する対象）
- 次のアクション: [具体的な次のステップ]
```
