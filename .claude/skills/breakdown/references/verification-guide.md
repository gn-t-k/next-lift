# 検証方法の選択ガイド

葉タスクの完了条件を書く際、以下の優先順で検証方法を選ぶ。フィードバックループが短い方法ほど、問題の早期発見につながる。

## 優先順位

1. **型システム** — コンパイル時に検証できる（最速のフィードバック）
2. **自動テスト** — 実行して振る舞いを検証する
3. **手動確認** — 自動化できない場合の最終手段

## 型システムで検証すべきケース

型エラーがないこと自体が完了の証拠になるケース:

- インターフェース・型定義の作成（例: APIレスポンスの型、propsの型）
- 既存の型に準拠する実装（例: ミドルウェアの型シグネチャ）
- 型の整合性が主目的の変更（例: ジェネリクスの追加、union型の拡張）
- 設定ファイルの型安全化（例: 環境変数のスキーマ定義）

完了条件の書き方: `型: [何が][どの型に]準拠すること`

## 自動テストで検証すべきケース

振る舞いの正しさを確認する必要があるケース:

- ビジネスロジック（例: 計算、バリデーション、フィルタリング）
- 状態遷移（例: フォームの状態管理、認証フロー）
- エラーハンドリング（例: 異常系の分岐、エラーメッセージの出力）
- データ変換（例: APIレスポンスからUIモデルへの変換）
- 条件分岐が複数ある処理

完了条件の書き方: `テスト: [どの条件で][何が起こる]こと`

## 手動確認が必要なケース

自動化のコストが検証の価値に見合わないケース:

- UIの見た目（例: レイアウト、色、フォント、レスポンシブ表示）
- アニメーション・トランジション
- 外部サービスとの連携（サンドボックス環境での動作確認）
- ブラウザ固有の挙動（例: Safari特有のCSS問題）
- アクセシビリティのスクリーンリーダー動作

完了条件の書き方: `確認: [何を][どう確認する]`

## 判断に迷うケースの指針

- 型とテストの両方が適用可能 → 型を優先（コストが低い）
- テストと手動確認の両方が適用可能 → テストを優先（再現可能）
- 1つのタスクに複数の検証方法が必要 → それぞれの完了条件を併記する
