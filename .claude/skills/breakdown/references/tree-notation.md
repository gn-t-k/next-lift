# タスクツリーの記法

## ノードのマーク

- `[?]` — 未展開ノード: さらに分解が必要。次のラウンドで具体化する対象
- `[ ]` — 葉タスク: TDDで着手可能な粒度まで具体化済み
- `[x]` — 完了タスク: 実装とテストが完了

## ツリーの構造

Markdownのネストされたリストで表現する。番号は階層を表す。

```markdown
### 1. 認証機能

- [?] 1.1. ログイン処理
- [ ] 1.2. セッション管理のミドルウェア作成
  - 説明: リクエストごとにセッションを検証するミドルウェアを作成する
  - 完了条件:
    - テスト: 有効なセッションでリクエストが通過すること
    - テスト: 無効なセッションで401が返ること
    - 型: ミドルウェアがNextRequestを受け取りNextResponseを返す型に準拠すること

### 2. ユーザープロフィール

- [?] 2.1. プロフィール取得API
- [?] 2.2. プロフィール編集画面
```

## 葉タスクの記述ルール

葉タスクには必ず「説明」と「完了条件」を記述する。

### 説明

何をするかを1〜2文で書く。実装方法の詳細は書かない（TDDで決める）。

### 完了条件

検証方法ごとに箇条書きで記述する:

- `テスト:` — 自動テストで検証する内容
- `型:` — 型システムで検証する内容
- `確認:` — 手動確認が必要な内容（自動化できない場合のみ）

## 幅優先の具体化ルール

1. タスクツリーを上から走査し、最初の `[?]` ノードを見つける
2. そのノードをサブタスクに分解する
3. サブタスクが葉タスクの粒度なら `[ ]`、まだ大きければ `[?]` にする
4. 1つのノードの分解が終わったら、次の `[?]` ノードに進む
5. 同一階層のすべての `[?]` が解消されてから、次の階層の `[?]` に進む
