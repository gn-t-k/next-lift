# Requirements - 要件定義ヒアリング・調査スキル

ユーザーの要望を深掘りし、調査を経て実装可能な要件にまとめる。

## 原則

- **前提を疑う**: ユーザーの「〜でいい」は未検討の裏返しであることが多い。暗黙の前提を放置すると実装後の手戻りコストが跳ね上がるため、妥当に見える仮定こそ重点的に問う
- **深さ優先**: 表層的な質問の繰り返しでは本質的な要件に辿り着けない。1つのトピックを2段階以上深掘りしてから次に進むことで、設計判断に必要な粒度の情報を引き出す
- **要望と要求を分離**: ユーザーは要望（実現したい状況）と要求（手段の提案）を混同しがち。要求に縛られると、より良い代替案を見落とす。要望を起点に最適解を探る
- **具体的な選択肢**: 自由記述の質問はユーザーの認知負荷が高く、議論が発散する。2〜4個の具体的な選択肢にpros/consを付けることで、判断を促しつつ議論を収束させる
- **調査に基づく提案**: 推測ベースの要件は実装フェーズで覆される。コードベースやWebの調査結果に基づく提案は、実現可能性が裏付けられているため手戻りが少ない
- **都度記録**: コンテキストウィンドウは有限で、長い議論ではコンテキストクリアにより内容が失われる。議論内容をファイルに書き込むことで、一貫性を保って継続できる

## 開始時

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力ファイル名はデフォルトで `requirements.md` とする
3. 呼び出し元から出力先ディレクトリやファイル名が指定された場合は、その指定に従う
4. 出力先に出力ファイルを `references/output-template.md` に基づいて作成する

## フェーズ1: 初期ヒアリング

### 目的

ユーザーの要望と要求を把握し、整理する。

### 手順

1. ユーザーの発言から要望（Goals）と要求（Requests）を抽出する
2. AskUserQuestionで不明点を確認する（1ラウンド2〜3問まで）
3. 要望と要求の対応関係を整理する
4. `requirements.md` の「要望」「要求」セクションを更新する

### ヒアリングのルール

- 各質問に2〜4個の具体的な選択肢を付ける（pros/consを簡潔に添える）
- 「他にありますか？」のようなオープンな質問は避ける
- 要望の裏にある動機を探る（「なぜそれが必要ですか？」）
- ユーザーの要求が要望の最適解とは限らないことを意識する
- 質問の具体例は `references/question-examples.md` を参照

### 完了条件

- 要望が1つ以上明確になっている
- 各要望に対する要求（またはまだない旨）が整理されている

## フェーズ2: 前提の特定

### 開始条件

- フェーズ1の完了条件を満たしている

### 目的

ユーザーの要望・要求・既存コードに潜む暗黙の前提を体系的に洗い出し、調査の優先順位を決める。

### 手順

1. フェーズ1の情報と既存コードから、暗黙の前提を抽出する
2. `references/assumption-categories.md` の8カテゴリで分類する
3. 各前提にリスクレベル（高/中/低）を付与する
4. `requirements.md` の「前提と検証」セクションに記録する

### 完了条件

- 前提が3つ以上特定されている
- 各前提にリスクレベルが付与されている
- 高リスクの前提がフェーズ3で優先的に調査されるよう整理されている

## フェーズ3: 調査

### 開始条件

- フェーズ2の 完了条件 を満たしている

### 目的

要望・要求の実現可能性を評価するための情報を集める。

### 手順

1. 高リスクの前提から優先的に調査する
2. コードベースを調査する
   - Agentツール（subagent_type: Explore）でコードベース調査を委譲する
   - 調査対象: 関連する既存実装、影響範囲、技術的制約、ADRとの整合性
   - 複数の調査対象がある場合は、Exploreエージェントを並列で起動する
3. 必要に応じてWebを調査する
   - WebSearch, WebFetch, context7 MCPツールを活用する
   - 調査対象: ライブラリのドキュメント、類似の実装パターン、APIの仕様
4. 調査結果を `requirements.md` の「調査結果」セクションに記録する
5. 前提の検証結果を「前提と検証」セクションに反映する
6. 実現可能性を評価し、表にまとめる

### 調査の記録基準

`requirements.md` に書く情報:
- 調査を経て得た結論（再調査コストが高い情報）
- ユーザーとの議論で合意した判断

書かなくてよい情報:
- 1回の検索で得られる事実（バージョン番号、APIシグネチャ等）
- コードを読めばわかる詳細（陳腐化リスクが高い）

## フェーズ4: 深掘り

### 開始条件

- フェーズ3で調査結果が記録されている

### 目的

調査結果に基づき、要件の詳細を詰める。

### 手順

1. 調査結果をユーザーに共有し、実現可能性の評価を伝える
2. AskUserQuestionで判断が必要な点を確認する
   - トレードオフの選択
   - スコープの調整
   - 代替案の検討
3. 新たに生じた疑問があれば追加調査する
4. `requirements.md` の「議論ログ」セクションにラウンドを追記する

### 掘り下げ戦略

各ラウンドで以下を実行する:

1. 回答が明らかにした新しい前提を特定する
2. 最も重要な未検証の前提を次ラウンドで追跡する
3. 1つのトピックにつき2段階以上深掘りする（表層的な回答で終わらせない）
4. 「前提と検証」セクションの未検証項目を確認し、調査が必要なものを特定する

### 完了条件（完了性の評価）

以下をすべて満たすまでフェーズ3〜4を繰り返す:

- [ ] 高リスクの前提がすべて検証済み
- [ ] 各主要トピックで2段階以上の深掘り完了
- [ ] トレードオフが明示的に記録されている（何を犠牲にしたかも含む）
- [ ] 要望と要件の間にギャップがない
- [ ] 保留事項が適切に記録されている

## フェーズ5: 要件確定

### 開始条件

- フェーズ4の 完了条件 を満たしている

### 目的

要件を確定し、ユーザーの合意を得る。

### 手順

1. 議論と調査の結果を要件（機能要件・非機能要件）としてまとめる
2. スコープ外の項目を明示する
3. 保留事項（今決めなくてよい判断）を整理し、決定タイミングを明記する
4. `requirements.md` の「要件」「スコープ外」「保留事項」セクションを更新する
5. ユーザーに最終確認を求める
6. 合意が得られたら「合意事項」のチェックボックスを更新する

### 完了条件

- すべての要望に対応する要件が定義されている
- 実現可能性が調査で裏付けられている
- スコープ外が明示されている
- 保留事項が整理され、決定タイミングが明記されている
- ユーザーが承認している

## フェーズの戻り条件

フェーズは基本的に前進するが、以下の条件で前のフェーズに戻る:

- → **フェーズ1**: 調査や深掘りの結果、要望自体が変わった・新たな要望が出た場合
- → **フェーズ2**: 深掘りや調査で新たな前提が見つかり、前提テーブルの再整理が必要な場合
- → **フェーズ3-4**: フェーズ4の完了性チェックリストを満たさない場合（既定の反復）
- → **フェーズ3-4**: フェーズ5の最終確認でスコープや要件の再検討が必要になった場合

## ラウンド終了時のステータスサマリー

各ラウンド（ヒアリング・調査・深掘り）の終了時に、以下のサマリーをユーザーに表示する。ユーザーが現在地と残作業を把握でき、フィードバックの質が上がるため。

```
**ステータス**
- 現在のフェーズ: X/5（フェーズ名）
- 確定済み要件: N件
- 未検証の前提: N件（うち高リスク: N件）
- 次のアクション: [具体的な次のステップ]
```
