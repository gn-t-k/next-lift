# Requirements - 要件定義ヒアリング・調査スキル

ユーザーの要望を深掘りし、調査を経て実装可能な要件にまとめる。

## 原則

- **深さ優先**: 広く浅くではなく、1つのトピックを2段階以上深掘りしてから次に進む
- **要望と要求を分離**: 要望（実現したい状況）と要求（ユーザーが提案する手段）を区別し、要求に縛られず最適解を探る
- **具体的な選択肢**: 自由記述ではなく、2〜4個の具体的な選択肢を提示して判断を促す
- **調査に基づく提案**: 推測ではなく、コードベースやWebの調査結果に基づいて要件を提案する
- **都度記録**: 議論内容をファイルに書き込み、コンテキストのコンパクション後も継続可能にする

## 開始時

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力先に `01-requirements.md` を `references/output-template.md` に基づいて作成する
3. dev-workflowから呼ばれた場合は、dev-workflowが指定するディレクトリを使う

## フェーズ1: 初期ヒアリング

### 目的

ユーザーの要望と要求を把握し、整理する。

### 手順

1. ユーザーの発言から要望（Goals）と要求（Requests）を抽出する
2. AskUserQuestionで不明点を確認する（1ラウンド2〜3問まで）
3. 要望と要求の対応関係を整理する
4. `01-requirements.md` の「要望」「要求」セクションを更新する

### ヒアリングのルール

- 各質問に2〜4個の具体的な選択肢を付ける（pros/consを簡潔に添える）
- 「他にありますか？」のようなオープンな質問は避ける
- 要望の裏にある動機を探る（「なぜそれが必要ですか？」）
- ユーザーの要求が要望の最適解とは限らないことを意識する

### 完了条件

- 要望が1つ以上明確になっている
- 各要望に対する要求（またはまだない旨）が整理されている

## フェーズ2: 調査

### 目的

要望・要求の実現可能性を評価するための情報を集める。

### 手順

1. コードベースを調査する（Grep, Glob, Read, Exploreエージェント）
   - 関連する既存実装
   - 影響範囲
   - 技術的制約
2. 必要に応じてWebを調査する（WebSearch, WebFetch, MCPツール）
   - ライブラリのドキュメント
   - 類似の実装パターン
   - APIの仕様
3. 調査結果を `01-requirements.md` の「調査結果」セクションに記録する
4. 実現可能性を評価し、表にまとめる

### 調査の記録基準

`01-requirements.md` に書く情報:
- 調査を経て得た結論（再調査コストが高い情報）
- ユーザーとの議論で合意した判断

書かなくてよい情報:
- 1回の検索で得られる事実（バージョン番号、APIシグネチャ等）
- コードを読めばわかる詳細（陳腐化リスクが高い）

## フェーズ3: 深掘り

### 目的

調査結果に基づき、要件の詳細を詰める。

### 手順

1. 調査結果をユーザーに共有し、実現可能性の評価を伝える
2. AskUserQuestionで判断が必要な点を確認する
   - トレードオフの選択
   - スコープの調整
   - 代替案の検討
3. 新たに生じた疑問があれば追加調査する
4. `01-requirements.md` の「議論ログ」セクションにラウンドを追記する

### 反復条件

以下のいずれかに該当する場合、フェーズ2〜3を繰り返す:

- 未解決の疑問や不確定要素がある
- 新しい調査が必要な論点が出た
- 要望と要件の間にギャップがある

## フェーズ4: 要件確定

### 目的

要件を確定し、ユーザーの合意を得る。

### 手順

1. 議論と調査の結果を要件（機能要件・非機能要件）としてまとめる
2. スコープ外の項目を明示する
3. `01-requirements.md` の「要件」「スコープ外」セクションを更新する
4. ユーザーに最終確認を求める
5. 合意が得られたら「合意事項」のチェックボックスを更新する

### 完了条件

- すべての要望に対応する要件が定義されている
- 実現可能性が調査で裏付けられている
- スコープ外が明示されている
- ユーザーが承認している

## コンパクション後の再開

コンテキストがコンパクションされた場合:

1. `01-requirements.md` を読み、現在の状況を把握する
2. 「議論ログ」の最新ラウンドを確認し、中断箇所を特定する
3. 未完了のフェーズから再開する
