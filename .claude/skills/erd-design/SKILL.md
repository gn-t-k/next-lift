# ERD設計スキル

要件からRDBのテーブル構造（ERD）を段階的に設計する。各ステップでMermaid ERDを出力し、ユーザーと合意しながら進める。

## 原則

- **イベント駆動で抽出する**: 「何が起きるか（イベント系）」から始める。「何が存在するか（リソース系）」はイベントの整理で自然に浮上する
- **段階的に進める**: 1ステップで完璧なERDを作らない。抽出→正規化→統合→導出整理→レビューと段階を踏む
- **各ステップでユーザーと合意する**: ERDは要件知識に依存する。各ステップでMermaid ERDとセルフレビュー結果を提示し、確認を得てから次に進む
- **都度記録する**: 設計の進捗と判断をファイルに書き込み、コンテキストクリアに備える
- **質問は具体的にする**: 各ステップでユーザーに確認する際は最大2〜3問に絞り、具体的な選択肢と pros/cons を提示する
- **修正は現ステップ内で行う**: ユーザーから指摘を受けた場合は現在のステップ内で修正する。前ステップの前提が崩れる場合はそのステップまで戻り、ERDを修正してから再度進める

## 開始時

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力ファイル名はデフォルトで `erd-design.md` とする
3. `references/output-template.md` に基づいてファイルを作成する
4. Explore エージェントで既存のコードベースを調査する:
   - `docs/architecture-decision-record/` からDB設計に関連するADRを検索
   - 既存のスキーマ定義（Drizzle schema等）を検索し、命名規則・既存テーブル構造を把握
   - 調査結果を出力ファイルの「要件サマリー」に記録する

## 各ステップ共通の手順

各ステップ（1〜4）の固有作業を行った後、以下を実施する:

1. Mermaid ERDを更新し、出力ファイルに書き込む
2. セルフレビュー（各リファレンス末尾のチェックリスト）を実施する
3. セルフレビュー結果に問題がなければERDをユーザーに提示し、確認を得る。問題があればAIの判断で修正し、修正後に提示する

## ステップ1: エンティティ抽出

要件からイベント系・リソース系エンティティを洗い出し、テーブル候補を特定する。

1. ユーザーから要件の入力を受け取る（画面仕様、ユースケース、既存データソース等）
2. 不明点があればAskUserQuestionで確認する
3. `references/step1-entity-extraction.md` を参照し、エンティティを抽出する

**完了条件**: イベント系・リソース系エンティティが網羅的に抽出され、ユーザーが承認している

## ステップ2: 正規化

`references/step2-normalization.md` を参照し、データの重複を排除してテーブル構造を整理する。

**完了条件**: データの重複が排除され、新たに浮上したリソース系エンティティが追加され、ユーザーが承認している

## ステップ3: リソース統合とリレーションシップ整理

`references/step3-resource-integration.md` を参照し、リソース系エンティティの重複を統合し、多重度とサブセットの扱いを確定する。

**完了条件**: リソースの統合が完了し、すべての多重度とサブセットの扱いが確定し、ユーザーが承認している

## ステップ4: 導出項目の整理

`references/step4-derived-cleanup.md` を参照し、導出項目の保持/排除を判断する。

**完了条件**: 導出項目の保持/排除が判断され、排除した項目の代替手段が決定し、ユーザーが承認している

## ステップ5: 最終レビュー

`references/step5-final-review.md` を参照し、ERD全体の整合性を確認する。開始時に調査した既存スキーマとの一貫性もチェックする。

**完了条件**: レビュー観点にすべて合格し、設計判断サマリーが記録され、ユーザーが最終承認している

## ラウンド終了時のステータスサマリー

各ラウンドの終了時に以下をユーザーに表示する:

```
**ステータス**
- 現在のステップ: X/5（ステップ名）
- エンティティ数: イベント系 N / リソース系 N
- 次のアクション: [具体的な次のステップ]
```
