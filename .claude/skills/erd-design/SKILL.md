# ERD設計スキル

要件からRDBのテーブル構造（ERD）を段階的に設計する。各フェーズをサブエージェントに委任し、メインエージェントはフェーズ進行とユーザーとの対話に集中する。

## アーキテクチャ

```
メインエージェント（このファイルに従う）
  ├── フェーズ進行の管理
  ├── ユーザーとの対話（レビュー、質問への回答）
  └── サブエージェント起動（フェーズごと）
        ├── 出力ファイルから現状を読み取り
        ├── 該当フェーズのreferenceを読み取り（他フェーズは読まない）
        ├── フェーズの作業を実行（セルフレビュー込み）
        └── 出力ファイルに結果を書き戻し
```

**重要: メインエージェントは `references/` 配下のフェーズファイルを直接読まない。** フェーズの実行はサブエージェントに委任する。

## 原則

- **段階的に進める**: 1フェーズで完璧なERDを作らない。抽出→正規化→表現方法の設計→関係の設計→検証と段階を踏む
- **レビュータイミングを分ける**: フェーズ1はユーザーレビュー必須。フェーズ2〜4はサブエージェントが完走し、確認事項があればメインエージェントがユーザーに提示。フェーズ5でユーザーが最終レビューする
- **都度記録する**: 設計の進捗と判断を出力ファイルに書き込み、コンテキストクリアに備える
- **質問は具体的にする**: ユーザーに確認する際は具体的な選択肢と pros/cons を提示する
- **修正は現フェーズ内で行う**: ユーザーから指摘を受けた場合はサブエージェントを再起動して修正する。前フェーズの前提が崩れる場合はそのフェーズまで戻る
- **後のフェーズの関心事はパーキングロットに記録する**: 現フェーズの範囲外の指摘は出力ファイルの「パーキングロット」に記録し、該当フェーズで対応する

## 開始時

メインエージェントが以下を実施する:

1. 出力先ディレクトリをAskUserQuestionで確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力ファイル名はデフォルトで `erd-design.md` とする
3. `references/output-template.md` に基づいてファイルを作成する
4. 外部サービス依存の事前確認: 以下の領域について、利用する外部サービスが決まっているかAskUserQuestionで確認する。決まっていればインプットとして記録し、決まっていなければフェーズ1の前に決定する:
   - **認証**: 認証ライブラリ/サービス（例: Better Auth, NextAuth, Clerk）→ 管理されるテーブル（user, session等）がERDの前提になる
   - **決済**: 決済サービス（例: Stripe, PayPay）→ 保持すべきID体系（payment_intent_id等）が決まる
   - **ファイルストレージ**: ストレージサービス（例: S3, Cloudflare R2）→ URLの持ち方が決まる
   - **メール/通知**: 通知サービス（例: SendGrid, FCM）→ 通知テーブルの設計に影響する
   - その他、テーブル構造に影響する外部依存があれば確認する
5. Explore エージェントで既存のコードベースを調査する:
   - `docs/architecture-decision-record/` からDB設計に関連するADRを検索
   - 既存のスキーマ定義（Drizzle schema等）を検索し、命名規則・既存テーブル構造を把握
   - 調査結果を出力ファイルの「要件サマリー」に記録する

## サブエージェント起動テンプレート

各フェーズでサブエージェント（general-purpose）を起動する際、以下のプロンプトテンプレートを使用する。`{...}` はメインエージェントが埋める。

```
ERD設計のフェーズ{N}を実行してください。

## 読むべきファイル
- 共通ルール: references/general-rules.md（全フェーズ共通の設計プロセスルール）
{フェーズ2以降の場合のみ: - テーブル設計ルール: references/table-design-rules.md（テーブル設計の制約とセルフレビューチェックリスト）}
- フェーズリファレンス: {referenceパス}（このフェーズの作業手順とセルフレビューチェックリスト）
- 出力ファイル: {出力ファイルパス}（現在の設計状態。ここに結果を書き戻す）
- 他のフェーズのリファレンスは読まないでください

## やること
1. 出力ファイルを読み、現在の設計状態と設計判断を把握する
2. パーキングロットにこのフェーズに該当する項目があれば対応する
3. リファレンスの作業手順に従ってフェーズの作業を実行する
4. 共通ルールとテーブル設計ルール（該当フェーズのみ）を必ず守る
5. 出力ファイルの該当フェーズセクションにMermaid ERDがある場合は更新する
6. リファレンス末尾のチェックリストとルールファイルのチェックリストの両方で自己検証し、問題があれば修正する
7. 出力ファイルの「ステータス」と「設計判断サマリー」を更新する

## 判断に迷う箇所の扱い
- 要件知識が必要で判断できない箇所は、暫定判断を置いて「⚠️ 要確認」マークをつける
- 暫定判断には理由（pros/cons）を記載する
- フェーズの作業は最後まで完走する（途中で止めない）

## 返却内容
作業完了後、以下をテキストで返してください:
1. このフェーズで行った変更のサマリー
2. 設計判断の一覧（判断内容と理由）
3. ユーザー確認が必要な項目（「⚠️ 要確認」の一覧）
{追加の指示があればここに記載}
```

## フェーズ1: エンティティの抽出

**リファレンス**: `references/phase1-entity-extraction.md`

1. サブエージェントを起動する
2. サブエージェントの結果を受けて、エンティティ一覧と設計判断をユーザーに提示する
3. 「⚠️ 要確認」項目があれば提示するが、**それとは別に**エンティティ一覧全体に対するフィードバックを求める。「⚠️ 要確認」項目への回答だけでフェーズ完了としない
4. ユーザーから明示的な承認（「OK」「進めて」等）を得てからフェーズ2に進む
5. 修正が必要な場合、ユーザーのフィードバックを追加の指示としてサブエージェントを再起動する

## フェーズ2: 正規化

**リファレンス**: `references/phase2-table-structure.md`

1. サブエージェントを起動する
2. サブエージェントの結果を確認する
3. 「⚠️ 要確認」項目があればユーザーに提示する
4. 確認事項がなければフェーズ3に進む。修正が必要ならサブエージェントを再起動する

## フェーズ3: 表現方法の設計

**リファレンス**: `references/phase3-representation.md`

1. サブエージェントを起動する
2. サブエージェントの結果を確認する
3. 「⚠️ 要確認」項目があればユーザーに提示する
4. 確認事項がなければフェーズ4に進む。修正が必要ならサブエージェントを再起動する

## フェーズ4: 関係の設計

**リファレンス**: `references/phase4-relationship.md`

1. サブエージェントを起動する
2. サブエージェントの結果を確認する
3. 「⚠️ 要確認」項目があればユーザーに提示する
4. 確認事項がなければフェーズ5に進む。修正が必要ならサブエージェントを再起動する

## フェーズ5: 検証

**リファレンス**: `references/phase5-final-review.md`

1. サブエージェントを起動する
2. サブエージェントの結果を受けて、全フェーズの設計判断サマリーと最終ERDをユーザーに提示する
3. ユーザーから最終承認を得る
4. 修正が必要な場合、該当フェーズのサブエージェントを再起動する

## ラウンド終了時のステータスサマリー

各ラウンドの終了時に以下をユーザーに表示する:

```
**ステータス**
- 現在のフェーズ: X/5（フェーズ名）
- エンティティ数: イベント系 N / リソース系 N
- 次のアクション: [具体的な次のフェーズ]
```

## 途中から再開する場合

ユーザーから途中再開の指示を受けたら `references/context-clear-recovery.md` を参照し、出力ファイルの現在の状態を把握してから作業を再開する。
