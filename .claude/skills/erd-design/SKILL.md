# ERD設計スキル

要件からRDBのテーブル構造（ERD）を段階的に設計する。各ステップでMermaid ERDを出力し、ユーザーと合意しながら進める。

## 原則

- **段階的に進める**: 1ステップで完璧なERDを作らない。抽出→正規化→統合→導出整理→レビューと段階を踏む
- **レビュータイミングを分ける**: ステップ1（エンティティ抽出）はユーザーレビュー必須。ステップ2〜4はAIセルフレビューで進行し、不明点のみAskUserQuestionで確認する。ステップ5でユーザーが最終レビューする
- **都度記録する**: 設計の進捗と判断をファイルに書き込み、コンテキストクリアに備える
- **質問は具体的にする**: 各ステップでユーザーに確認する際は最大2〜3問に絞り、具体的な選択肢と pros/cons を提示する
- **修正は現ステップ内で行う**: ユーザーから指摘を受けた場合は現在のステップ内で修正する。前ステップの前提が崩れる場合はそのステップまで戻り、ERDを修正してから再度進める
- **後のステップの関心事はパーキングロットに記録する**: ユーザーから現在のステップの範囲外の指摘を受けた場合、「パーキングロット（後で扱う項目）」として出力ファイルに記録し、該当ステップで確実に対応する旨を伝える。ただし、現ステップの前提を覆す指摘は即座に対応する

## 開始時

1. 出力先ディレクトリをユーザーに確認する（例: `docs/development/[yyyy-mm-dd-feature-name]/`）
2. 出力ファイル名はデフォルトで `erd-design.md` とする
3. `references/output-template.md` に基づいてファイルを作成する
4. 外部サービス依存の事前確認: 以下の領域について、利用する外部サービスが決まっているかユーザーに確認する。決まっていればインプットとして記録し、決まっていなければステップ1の前に決定する:
   - **認証**: 認証ライブラリ/サービス（例: Better Auth, NextAuth, Clerk）→ 管理されるテーブル（user, session等）がERDの前提になる
   - **決済**: 決済サービス（例: Stripe, PayPay）→ 保持すべきID体系（payment_intent_id等）が決まる
   - **ファイルストレージ**: ストレージサービス（例: S3, Cloudflare R2）→ URLの持ち方が決まる
   - **メール/通知**: 通知サービス（例: SendGrid, FCM）→ 通知テーブルの設計に影響する
   - その他、テーブル構造に影響する外部依存があれば確認する
5. Explore エージェントで既存のコードベースを調査する:
   - `docs/architecture-decision-record/` からDB設計に関連するADRを検索
   - 既存のスキーマ定義（Drizzle schema等）を検索し、命名規則・既存テーブル構造を把握
   - 調査結果を出力ファイルの「要件サマリー」に記録する

## 各ステップ共通の手順

各ステップの固有作業を行った後、以下を実施する:

0. パーキングロットに現ステップに該当する項目がないか確認し、あれば本ステップで対応する
1. Mermaid ERDを更新し、出力ファイルに書き込む
2. セルフレビュー: Exploreエージェントを起動し、出力ファイルの現ステップの内容と該当リファレンス末尾のチェックリストを渡してレビューを依頼する。問題があればAIの判断で修正する
3. レビュー:
   - **ステップ1**: ERDと設計判断をユーザーに提示し、承認を得てから次に進む
   - **ステップ2〜4**: AIセルフレビューで進行する。要件知識が必要な不明点のみAskUserQuestionで確認する。セルフレビュー結果と設計判断を出力ファイルに記録し、次のステップに進む
   - **ステップ5**: 全ステップの設計判断サマリーと最終ERDをユーザーに提示し、最終承認を得る

各ラウンドの終了時に以下をユーザーに表示する:

```
**ステータス**
- 現在のステップ: X/5（ステップ名）
- エンティティ数: イベント系 N / リソース系 N
- 次のアクション: [具体的な次のステップ]
```
