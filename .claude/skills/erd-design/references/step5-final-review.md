# ステップ5: 最終レビュー

## このステップで何をするか

- **ゴール**: ステップ1〜4で作成したERD全体を見直し、設計上の問題がないか確認する。よくある間違いと照合し、最終的なERDをユーザーと合意する
- **インプット**: ステップ4までの最終ERDと設計判断の記録
- **アウトプット**: レビュー済みの最終ERD、設計判断サマリー

## なぜ最終レビューが必要なのか

ステップ1〜4は各ステップの目的に集中しているため、全体を俯瞰したときに初めて見える問題がある。このステップでは個別の判断ではなく、ERD全体の整合性と品質を確認する。

## レビュー観点

### 1. よくある設計の間違い

以下に該当しないか、ERD全体を確認する:

**過剰に複雑な設計になっていないか**

あらゆるケースを想定して重厚なデータモデルを作ってしまうと、それを扱うアプリケーションも複雑になる。要件に対して過剰な構造がないか確認する。

- 現時点の要件に不要なテーブルや属性がないか
- 「将来必要になるかもしれない」だけで追加したテーブルがないか
- DBだけで解決しようとせず、プログラムとの役割分担を考えているか

**判断基準をパターンとして暗記していないか**

ステップ1〜4で学んだパターン（ヘッダ・ディテール、中間テーブルなど）を理由なく機械的に適用していないか確認する。パターンは結果であり、出発点ではない。

- 「なぜこの構造にしたか」を各テーブルについて説明できるか
- 要件に合わないパターンを無理に適用していないか

**要件のヒアリング不足はないか**

ERDを描く過程で浮かんだ疑問点を、ユーザーに確認せず想定で埋めていないか確認する。

- 多重度の判断で「たぶん1:Nだろう」と想定した箇所はないか
- サブセットの判断で、将来の拡張についてユーザーに確認したか
- 導出項目の保持/排除で、ビジネスルールを正確に把握しているか

### 2. 構造の整合性チェック

| チェック項目 | 確認方法 |
|---|---|
| 孤立テーブルがないか | すべてのテーブルが少なくとも1つのリレーションシップを持つか |
| 循環参照がないか | A → B → C → A のような循環がないか |
| 全テーブルにサロゲートキー（ID）があるか | 業務コードが主キーになっていないか |
| 多重度が全リレーションシップに明示されているか | ERD上で曖昧な関係が残っていないか |
| 命名の一貫性 | テーブル名・カラム名の命名規則が統一されているか（複数形/単数形、snake_case等） |
| NULLableカラムが残っていないか | 全テーブルの全カラムについてNULL許容の正当化理由があるか |
| UPDATE管理のカラムに正当化理由があるか | 設計判断サマリーに記録されているか |

## セルフレビュー（AIチェック）

このステップの完了時にAIが確認する:

- [ ] 孤立テーブル・循環参照がないか
- [ ] 全テーブルにサロゲートキーが設定されているか
- [ ] 全リレーションシップに多重度が明示されているか
- [ ] 命名規則が統一されているか（複数形/単数形、snake_case等）
- [ ] 全ステップの設計判断とその理由が記録されているか
- [ ] 最終ERDがMermaid形式で最新の状態を反映しているか

## ユーザー確認事項

セルフレビューの結果、AIの判断だけでは確定できない問題が見つかった場合にユーザーに確認する。確認が必要な典型的なケース:

- 過剰に複雑な構造がないか（要件に対して適切な複雑さかはビジネス判断）
- 多重度を想定で埋めた箇所がないか（「たぶん1:Nだろう」と判断した箇所）
- サブセットの方針で将来の拡張について合意が取れていない箇所
- 未解決の疑問点

セルフレビューで問題が見つからなければ、ユーザーには「セルフレビューで問題なし」と報告し、最終承認を求める。
