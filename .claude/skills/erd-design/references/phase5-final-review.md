# フェーズ5: 最終レビュー

## このフェーズで何をするか

- **ゴール**: フェーズ1〜4で作成したERD全体を見直し、要件を実現できるか検証し、設計上の問題がないか確認する
- **インプット**: フェーズ4までの最終ERDと設計判断の記録、要件サマリー/ユースケース
- **アウトプット**: 要件カバレッジ検証済み・レビュー済みの最終ERD、設計判断サマリー

## なぜ最終レビューが必要なのか

フェーズ1〜4は各フェーズの目的に集中しているため、全体を俯瞰したときに初めて見える問題がある。特に、フェーズ2〜4で正規化・テーブル分割・関係の設計を行う過程でエンティティの統合・分割・属性の移動が起きるため、フェーズ1で対応していた要件が最終ERDでは表現できなくなっている可能性がある。

このフェーズでは個別の判断ではなく、ERD全体の整合性と品質を確認する。

## 作業手順

### ステップ1: 要件カバレッジの検証

要件サマリーおよびユースケースを1つずつ辿り、最終ERDのテーブル・カラムで実現できるかを擬似SQLで検証する。

#### なぜ擬似SQLで検証するのか

「このユースケースはERDで実現できる」という判断を、目視ではなく具体的なクエリとして証明する。JOINパスの不足、カラムの欠落、多重度の誤りなど、視覚的なレビューでは見落としやすい問題が顕在化する。ERDが「きれいに見える」のと「実際にデータを読み書きできる」は別問題。

#### 手順

**1-1. ユースケースを列挙する**

出力ファイルの「要件サマリー」と、要件ドキュメント（ユースケース定義、画面仕様など）から、ERDがサポートすべきユースケースを列挙する。

**1-2. 各ユースケースに対して擬似SQLを書く**

各ユースケースについて、以下の観点で擬似SQLを書く:

- **読み取り（SELECT）**: そのユースケースで表示・参照するデータを取得できるか。JOINパスが通るか、WHERE条件に使えるカラムがあるか
- **書き込み（INSERT）**: そのユースケースで発生するデータを保存できるか。必要なカラムが揃っているか

擬似SQLはテーブル名・カラム名を使った概念的なクエリで、厳密なSQL構文である必要はない。「どのテーブルをJOINするか」「どのカラムで絞り込むか」が分かればよい。

**1-3. 検証結果を記録する**

各ユースケースの擬似SQLと検証結果（OK / NG）を出力ファイルの「要件カバレッジ検証」テーブルに記録する。NGのユースケースは「⚠️ 要確認」としてユーザーに報告する。

### ステップ2: よくある設計の間違いチェック

以下に該当しないか、ERD全体を確認する。

**過剰に複雑な設計になっていないか**

あらゆるケースを想定して重厚なデータモデルを作ってしまうと、それを扱うアプリケーションも複雑になる。要件に対して過剰な構造がないか確認する。

- 現時点の要件に不要なテーブルや属性がないか
- 「将来必要になるかもしれない」だけで追加したテーブルがないか
- DBだけで解決しようとせず、プログラムとの役割分担を考えているか

**判断基準をパターンとして暗記していないか**

フェーズ1〜4で学んだパターン（ヘッダ・ディテール、中間テーブルなど）を理由なく機械的に適用していないか確認する。パターンは結果であり、出発点ではない。

- 「なぜこの構造にしたか」を各テーブルについて説明できるか
- 要件に合わないパターンを無理に適用していないか

**要件のヒアリング不足はないか**

ERDを描く過程で浮かんだ疑問点を、ユーザーに確認せず想定で埋めていないか確認する。

- 多重度の判断で「たぶん1:Nだろう」と想定した箇所はないか
- サブセットの判断で、将来の拡張についてユーザーに確認したか
- 導出項目の保持/排除で、ビジネスルールを正確に把握しているか

### ステップ3: 構造の整合性チェック

| チェック項目 | 確認方法 |
|---|---|
| 命名の一貫性 | テーブル名・カラム名の命名規則が統一されているか（複数形/単数形、snake_case等） |
| NULLableカラムの3点チェックが実施されているか | NULLable属性カラムについて3値論理・NULL伝播・パフォーマンスの検討記録があるか |
| UPDATE/DELETE管理の判断プロセスが踏まれているか | 将来の履歴要件シナリオの提示、ユーザーの確認、後方互換性の評価が設計判断に記録されているか |
| 既存スキーマとの一貫性 | 開始時に調査した既存テーブルの命名規則・構造と矛盾がないか |

## 具体例: ウォークスルー

### toC例: SNSアプリの最終レビュー

**要件**: ユーザーがテキスト・画像付きの投稿ができる。他のユーザーをフォローでき、投稿にいいね・コメントができる。

**ステップ1（要件カバレッジ検証）を適用:**

| # | ユースケース | 擬似SQL | 結果 |
|---|---|---|---|
| 1 | ユーザーがテキスト投稿する | `INSERT INTO posts (id, user_id, content, published_at) VALUES (...)` | OK |
| 2 | ユーザーが画像付き投稿する | `INSERT INTO posts (id, user_id, content, media_type, media_url, published_at) VALUES (...)` | OK |
| 3 | タイムラインを表示する（フォロー中の投稿一覧） | `SELECT p.* FROM posts p JOIN follows f ON f.followee_id = p.user_id WHERE f.follower_id = ? ORDER BY p.published_at DESC` | OK |
| 4 | 投稿にいいねする | `INSERT INTO likes (id, user_id, post_id, liked_at) VALUES (...)` | OK |
| 5 | 投稿のいいね数を表示する | `SELECT COUNT(*) FROM likes WHERE post_id = ?` | OK |
| 6 | 投稿にコメントする | `INSERT INTO comments (id, user_id, post_id, content, commented_at) VALUES (...)` | OK |
| 7 | 投稿のコメント一覧を表示する | `SELECT c.*, u.name FROM comments c JOIN users u ON u.id = c.user_id WHERE c.post_id = ? ORDER BY c.commented_at` | OK |
| 8 | ユーザーをフォローする | `INSERT INTO follows (id, follower_id, followee_id, followed_at) VALUES (...)` | OK |
| 9 | フォロワー一覧を表示する | `SELECT u.* FROM users u JOIN follows f ON f.follower_id = u.id WHERE f.followee_id = ?` | OK |
| 10 | 投稿にタグをつける | `INSERT INTO post_tags (id, post_id, tag_id) VALUES (...)` | OK |
| 11 | タグで投稿を検索する | `SELECT p.* FROM posts p JOIN post_tags pt ON pt.post_id = p.id JOIN tags t ON t.id = pt.tag_id WHERE t.name = ?` | OK |

**ステップ2（よくある間違いチェック）を適用:**
- 過剰な複雑さ: テーブル数7。要件に対して適切
- パターンの機械的適用: なし
- ヒアリング不足: フォロー解除の扱い（DELETE or イベント）がユーザー未確認 → ⚠️ 要確認

**ステップ3（構造の整合性チェック）を適用:**
- 孤立テーブル: なし ✓
- 循環参照: なし ✓
- サロゲートキー: 全テーブルにid設定済み ✓
- 多重度: 全リレーションシップに明示済み ✓
- 命名: snake_case統一 ✓

### toB例: EC受注管理の最終レビュー

**要件**: 顧客が商品を注文する。1回の注文で複数商品を購入でき、合計に送料と消費税が加算される。

**ステップ1（要件カバレッジ検証）を適用:**

| # | ユースケース | 擬似SQL | 結果 |
|---|---|---|---|
| 1 | 商品を注文する（複数商品） | `INSERT INTO orders (...) VALUES (...); INSERT INTO order_items (order_id, product_id, quantity, unit_price) VALUES (...), (...);` | OK |
| 2 | 注文一覧を表示する | `SELECT o.*, u.name FROM orders o JOIN users u ON u.id = o.user_id WHERE o.user_id = ? ORDER BY o.ordered_at DESC` | OK |
| 3 | 注文明細を表示する（小計含む） | `SELECT oi.*, oi.unit_price * oi.quantity AS subtotal, p.name FROM order_items oi JOIN products p ON p.id = oi.product_id WHERE oi.order_id = ?` | OK |
| 4 | 注文の合計金額（送料・税込み）を表示する | `SELECT total_amount, shipping_fee, tax FROM orders WHERE id = ?` | OK |
| 5 | 決済を記録する | `INSERT INTO payments (id, order_id, method, amount, paid_at) VALUES (...)` | OK |
| 6 | 出荷を記録する | `INSERT INTO shipments (id, order_id, tracking_number, shipped_at) VALUES (...)` | OK |
| 7 | カテゴリ別の商品一覧を表示する | `SELECT p.* FROM products p JOIN product_categories pc ON pc.id = p.category_id WHERE pc.name = ?` | OK |
| 8 | 配送先を選択して注文する | `SELECT * FROM shipping_addresses WHERE user_id = ?; INSERT INTO orders (..., shipping_address_id, ...) VALUES (...)` | OK |

**ステップ2（よくある間違いチェック）を適用:**
- 過剰な複雑さ: テーブル数8。要件に対して適切
- パターンの機械的適用: ヘッダ・ディテール（orders/order_items）は「合計に送料と税を加算」の要件に適合 ✓
- ヒアリング不足: 返品・返金の要件があるならテーブルが未設計 → ⚠️ 要確認

**ステップ3（構造の整合性チェック）を適用:**
- 孤立テーブル: なし ✓
- 循環参照: なし ✓
- サロゲートキー: 全テーブルにid設定済み ✓
- 多重度: 全リレーションシップに明示済み ✓
- 命名: snake_case統一 ✓

## セルフレビュー（AIチェック）

このフェーズの完了時にAIが確認する:

- [ ] すべての要件/ユースケースに対応する擬似SQLが書けることを確認したか
- [ ] 擬似SQLで読み取り（SELECT）と書き込み（INSERT）の両方を検証したか
- [ ] 命名規則が統一されているか（複数形/単数形、snake_case等）
- [ ] 全フェーズの設計判断とその理由が記録されているか
- [ ] 最終ERDがMermaid形式で最新の状態を反映しているか

## ユーザー確認事項

セルフレビューの結果、AIの判断だけでは確定できない問題が見つかった場合にユーザーに確認する。確認が必要な典型的なケース:

- 擬似SQLが書けないユースケースがあった場合（構造の不足）
- 過剰に複雑な構造がないか（要件に対して適切な複雑さかはビジネス判断）
- 多重度を想定で埋めた箇所がないか（「たぶん1:Nだろう」と判断した箇所）
- サブセットの方針で将来の拡張について合意が取れていない箇所
- 未解決の疑問点

セルフレビューで問題が見つからなければ、ユーザーには「セルフレビューで問題なし」と報告し、最終承認を求める。
