# フェーズ1: エンティティ抽出

## このフェーズで何をするか

- **ゴール**: システムが管理すべき「データのまとまり」を洗い出し、エンティティの候補を特定する
- **インプット**: 要件（画面仕様、ユースケース、既存のデータソースなど）
- **アウトプット**: エンティティの一覧（名前・種別・主要属性）

## 前提知識

### エンティティの2種類

エンティティは「システムが記録・管理する必要があるデータのまとまり」。

| | イベント系 | リソース系 |
|---|---|---|
| **何を表すか** | 「起きたこと」の記録 | 「存在するもの」の情報 |
| **特徴** | 動詞で表現できる。発生日時を持つ | 名詞で表現できる。名前を持つ |
| **例（EC）** | 注文、出荷、返品、決済 | ユーザー、商品、配送先 |
| **例（SNS）** | 投稿、いいね、フォロー、通報 | ユーザー、コミュニティ、タグ |

### イベント系かリソース系かの判別

迷ったら以下のテストを使う:

- 「〜する」が成立するか → イベント系（「注文する」✓ / 「ユーザーする」✗）
- 「〜日」が成立するか → イベント系（「注文日」✓ / 「ユーザー日」✗）
- 「〜名」が成立するか → リソース系（「ユーザー名」✓ / 「注文名」✗）

## 作業手順

以下のステップではイベント系から先に抽出する。理由:
- イベント系は要件（「ユーザーが何をするか」）から直接見つけやすい
- リソース系はイベントを整理する過程で自然に浮かび上がる（「誰が」「何を」「どこで」）
- リソースから始めると、実際には使われないエンティティを作りがち

### ステップ1: エンティティを洗い出す

要件からイベント系・リソース系エンティティを洗い出す。

1. ユーザーのアクションを列挙する（「ユーザーは何をするか？」「誰が関わるか？」「何に対して行うか？」「どんな種別・区分があるか？」）
2. 「〜する」「〜日」テストでイベント系を特定する
3. イベント系の「誰が」「何を」「どこで」からリソース系を浮上させる

### ステップ2: 属性を列挙する

以下の観点で各エンティティの主要な属性をリストアップする。網羅性を重視し、重複排除・精査はフェーズ2で行う。この時点では型（varchar, integerなど）は決めず、属性の観点だけを記録する。

| 観点 | 例 |
|---|---|
| いつ？（日時） | 作成日時, 注文日時, 公開日時 |
| いくつ？（数量） | 数量, 閲覧数, いいね数 |
| いくら？（金額） | 価格, 合計金額, 値引額 |
| なぜ？（理由・状態） | 理由, ステータス, キャンセル理由 |
| 誰が・何を？（参照） | ユーザー, 商品, 投稿 |

### ステップ3: イベントの漏れを検出する

ステップ2で列挙した属性を手がかりに、見落としたイベントがないか検出する。

**入力と出力の対を確認する**:
- 「入金」があるなら「出金」は？
- 「フォロー」があるなら「アンフォロー」は？
- 「注文」があるなら「キャンセル」は？

**状態遷移を追いかける**:
- EC: 注文 → 決済 → 出荷 → 配達完了 → （返品）
- SNS: 下書き保存 → 投稿 → （編集） → （削除）

各状態はそれぞれ独立した事実（「起きたこと」）なので、個別のイベント系エンティティとして識別する。1つのエンティティに複数の状態を混ぜると、状態ごとに必要な属性を見落とす。

**「あるときだけ値が入る」属性から隠れたイベントを見つける**:
- 既読日時（未読のときは値がない） → 「既読」イベントが隠れている
- キャンセル日時（キャンセルしていなければ値がない） → 「キャンセル」イベントが隠れている
- 削除日時（削除されていなければ値がない） → 「削除」イベントが隠れている

**前後に隠れたイベントを想像する**:
- 「予約」があるなら、後続に「予約の実行（チェックイン等）」があるはず
- 「訂正」があるなら、先行に「訂正対象を作ったイベント」があるはず

## このフェーズでやらないこと

このフェーズはエンティティと属性の発見に集中する。判断の理由はエンティティレベル（「これは別の事実として記録すべきか」「このデータは誰が・いつ・何に対して発生させるか」）で語る。

以下はフェーズ2以降で扱うため、このフェーズでは判断しない:

- **テーブル構造の設計判断をしない**: NULL回避、UPDATE/DELETE回避、enum/CHECK、正規化などの判断はしない。テーブルをどう構成するかはフェーズ2・3で扱う
- **リレーションシップの多重度を厳密に決めない**: エンティティ間の関係はフェーズ4で設計する

## 具体例: ウォークスルー

### toC例: SNSアプリ

**要件**: ユーザーがテキスト・画像付きの投稿ができる。他のユーザーをフォローでき、投稿にいいね・コメントができる。

**ステップ1: エンティティを洗い出す**
- アクション: 投稿する、いいねする、コメントする、フォローする、プロフィールを編集する
- イベント系: 投稿、いいね、コメント、フォロー
- リソース系: ユーザー、タグ/カテゴリ

**ステップ2: 属性を列挙する**

| エンティティ | 種別 | 主要属性 |
|---|---|---|
| ユーザー | リソース | 名前, メール, アバターURL, 自己紹介 |
| 投稿 | イベント | 本文, 画像URL, 公開日時, ユーザー |
| いいね | イベント | 作成日時, ユーザー, 投稿 |
| コメント | イベント | 本文, 作成日時, ユーザー, 投稿 |
| フォロー | イベント | 作成日時, フォローする人, フォローされる人 |
| タグ | リソース | 名前 |

**ステップ3: 漏れを検出する**
- フォロー ↔ アンフォロー？ → フォローの削除で表現するか、別イベントにするかはユーザーに確認
- 投稿の編集・削除は？ → 要件次第で追加

### toB例: EC受注管理

**要件**: 顧客が商品を注文する。1回の注文で複数商品を購入でき、合計に送料と消費税が加算される。

**ステップ1: エンティティを洗い出す**
- アクション: 商品を注文する、決済する、出荷する、返品する
- イベント系: 注文、決済、出荷、返品
- リソース系: ユーザー、商品、商品カテゴリ、配送先

**ステップ2: 属性を列挙する**

| エンティティ | 種別 | 主要属性 |
|---|---|---|
| ユーザー | リソース | 名前, メール |
| 商品 | リソース | 名前, 価格, カテゴリ |
| 商品カテゴリ | リソース | 名前 |
| 配送先 | リソース | ユーザー, 住所, 郵便番号 |
| 注文 | イベント | ユーザー, 配送先, 合計金額, 送料, 税, 注文日時 |
| 注文明細 | イベント | 注文, 商品, 数量, 単価, 小計 |
| 決済 | イベント | 注文, 方法, 金額, 決済日時 |
| 出荷 | イベント | 注文, 追跡番号, 出荷日時 |

**ステップ3: 漏れを検出する**
- 注文 → 決済 → 出荷 → 配達完了の流れで「配達完了」が未抽出 → 要件次第で追加
- 返品があるなら「返金」は？ → ユーザーに確認

## セルフレビュー

このフェーズの完了時に以下を確認する:

- [ ] すべてのユースケース/画面に対応するイベントが抽出されているか
- [ ] 各イベント系エンティティに日時の属性があるか
- [ ] イベントの対（作成↔削除、入↔出）に漏れがないか
- [ ] 状態遷移が個別のイベント系エンティティとして識別されているか
- [ ] リソース系がイベント系の「誰が」「何を」から浮上しているか（リソースから始めていないか）
- [ ] 同じ概念を別名で重複抽出していないか
