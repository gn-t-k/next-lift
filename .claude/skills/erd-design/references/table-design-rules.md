# テーブル設計ルール（フェーズ2以降）

## テーブル設計の制約

- **enum/CHECK禁止**: RDBのenum型やcheck制約は使わない。値の追加にマイグレーションが必要で保守コストが高い。アプリ側のバリデーションで管理する
- **JOIN複雑性は理由にならない**: 「JOINが増えてクエリが複雑になる」は正規化やテーブル分割を避ける理由にならない。多少SQLが複雑になったとしてもAIエージェントを活用することで内容を把握できるし、テストを書いておくことで安全に改修できる。適切なindexを設定すればパフォーマンスも問題にならない。クエリが書けないレベルで複雑になる場合のみ考慮する
- **センチネル値禁止**: NULLを回避する手段として、空文字（`""`）・`0`・`-1`・`"N/A"`などのセンチネル値をデフォルト値として使わない。これらの値はドメイン上の正当な値と区別がつかず、NULLよりも深刻な問題を引き起こす:
  - 「値が未設定」と「ユーザーが空文字/ゼロを入力した」を区別できなくなる
  - NULLはSQL標準（IS NULL, COALESCE）で統一的に扱えるが、センチネル値は規約に依存するため、読み手全員がその規約を知らないとバグになる
  - NULLを避けるべき理由はテーブル構造で解決する（フェーズ3の代替手段: イベントテーブル分離・サブセットテーブル・中間テーブル）。カラムのデフォルト値で解決しようとしない
  - デフォルト値が適切なのは、その値がドメイン上の正当な初期値である場合のみ（例: `ordered_at DEFAULT NOW()`、`is_active DEFAULT true`）

- **タイムスタンプの命名**: 全エンティティに一律で`created_at`/`updated_at`を付けない。タイムスタンプカラムには、そのカラムが記録する事実に即した名前を付ける（例: `posted_at`, `reserved_at`, `paid_at`, `read_at`）。適切な名前が思いつかない場合は、そのカラムが本当に必要かを疑う。これはイベントエンティティにおいて特に重要で、イベントの日時属性の命名がエンティティの意味を明確にする
  - 参考: イミュータブルデータモデル https://scrapbox.io/kawasima/イミュータブルデータモデル の「すべてのエンティティに付ける登録日時や更新日時」セクション

## セルフレビューチェックリスト

フェーズ固有のチェックリストに加えて、以下を必ず確認する:

- [ ] enum型やCHECK制約を使っていないか
- [ ] 「JOINが複雑」を理由にテーブル分割を避けていないか
- [ ] NULLを回避するためにセンチネル値（空文字、0、-1等）をデフォルト値として使っていないか
- [ ] `created_at`や`updated_at`のような汎用的な名前のタイムスタンプがないか（事実に即した名前が付いているか）
