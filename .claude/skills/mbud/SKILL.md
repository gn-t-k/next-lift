# モデルベースUIデザインスキル

UIの構造設計を段階的に行う。見た目（モック）からではなく、ユースケース・タスク・概念モデルといった抽象的な設計基盤から積み上げるプロセスを管理する。

各フェーズをサブエージェントに委任し、メインエージェントはフェーズ進行とユーザーとの対話に集中する。

## アーキテクチャ

```
メインエージェント（このファイルに従う）
  ├── フェーズ進行の管理
  ├── ユーザーとの対話（レビュー、質問への回答）
  └── サブエージェント起動（フェーズごと）
        ├── 出力ファイルから現状を読み取り
        ├── 該当フェーズのreferenceを読み取り（他フェーズは読まない）
        ├── フェーズの作業を実行（セルフレビュー込み）
        └── 出力ファイルに結果を書き戻し
```

**重要: メインエージェントは `references/` 配下のフェーズファイルを直接読まない。** フェーズの実行はサブエージェントに委任する。

## 原則

- **段階的に進める**: 1フェーズで完璧なUI設計を作らない。抽出→分類→構造化→統合→検証と段階を踏む
- **レビュータイミングを分ける**: フェーズ1はユーザーレビュー必須。フェーズ2〜5は作業サブエージェントとレビューサブエージェントを分けて実行し、確認事項をメインエージェントが提示。フェーズ6でユーザーが最終レビュー
- **都度記録する**: 設計の進捗と判断を出力ファイルに書き込み、コンテキストクリアに備える
- **質問は具体的にする**: ユーザーに確認する際は具体的な選択肢とpros/consを提示する
- **修正は現フェーズ内で行う**: ユーザーからの指摘はサブエージェントを再起動して修正。前フェーズの前提が崩れる場合はそのフェーズまで戻る
- **パーキングロットを活用する**: 後続フェーズの関心事は出力ファイルの「パーキングロット」に記録し、該当フェーズで対応する

## 開始時

メインエージェントが以下を実施する:

1. 出力先ディレクトリをAskUserQuestionで確認する
2. 出力ファイル名はデフォルトで `ui-design.md` とする
3. `${CLAUDE_SKILL_DIR}/references/output-template.md` に基づいてファイルを作成する
4. 入力資料の確認: ペルソナモデルと行動シナリオの所在をユーザーに確認する
5. 対象プラットフォーム/フォームファクタの事前確認: どのデバイス環境を対象とするかAskUserQuestionで確認する
6. Exploreエージェントで既存のコードベースを調査する:
   - `docs/architecture-decision-record/` からUI設計に関連するADRを検索
   - 既存のUI設計資産があれば検索
   - 調査結果を出力ファイルの「要件サマリー」に記録する

## サブエージェント起動テンプレート

各フェーズでサブエージェント（general-purpose）を起動する際、以下のプロンプトテンプレートを使用する。`{...}` はメインエージェントが埋める。

```
モデルベースUIデザインのフェーズ{N}を実行してください。

## 読むべきファイル
- 共通ルール: ${CLAUDE_SKILL_DIR}/references/general-rules.md（全フェーズ共通の設計ルール）
- フェーズリファレンス: {referenceパス}（このフェーズの作業手順とチェックリスト）
- 出力ファイル: {出力ファイルパス}（現在の設計状態。ここに結果を書き戻す）
- 他のフェーズのリファレンスは読まないでください

## やること
1. 出力ファイルを読み、現在の設計状態と設計判断を把握する
2. パーキングロットにこのフェーズに該当する項目があれば対応する
3. リファレンスの作業手順に従ってフェーズの作業を実行する
4. 共通ルールを必ず守る
5. 出力ファイルの該当フェーズセクションに成果物を書き込む
6. 出力ファイルの「ステータス」と「設計判断ログ」を更新する

## レビューは別サブエージェントに委譲する

作業を実行するサブエージェントとレビューするサブエージェントは**必ず分ける**。作業サブエージェントの完了後、メインエージェントはレビュー用サブエージェントを起動し、以下を読ませてレビューさせる:

- `${CLAUDE_SKILL_DIR}/references/general-rules.md`（共通チェックリスト）
- 該当フェーズのリファレンス（フェーズ固有チェックリスト）
- 出力ファイル（レビュー対象）

レビューサブエージェントは問題を検出した場合、指摘事項を返す。メインエージェントは指摘を作業サブエージェントに渡して修正させる。

## 判断に迷う箇所の扱い
- 要件知識が必要で判断できない箇所は、暫定判断を置いて「⚠️ 要確認」マークをつける
- 暫定判断には理由（pros/cons）を記載する
- フェーズの作業は最後まで完走する（途中で止めない）

## 返却内容
作業完了後、以下をテキストで返してください:
1. このフェーズで行った変更のサマリー
2. 設計判断の一覧（判断内容と理由）
3. ユーザー確認が必要な項目（「⚠️ 要確認」の一覧）
{追加の指示があればここに記載}
```

## フェーズ1: ユースケース定義

**リファレンス**: `${CLAUDE_SKILL_DIR}/references/phase1-use-case-definition.md`

1. サブエージェントを起動する
2. サブエージェントの結果を受けて、ユースケース一覧と設計判断をユーザーに提示する
3. 「⚠️ 要確認」項目があれば提示するが、**それとは別に**ユースケース一覧全体に対するフィードバックを求める
4. ユーザーから明示的な承認を得てからフェーズ2に進む
5. 修正が必要な場合、フィードバックを追加の指示としてサブエージェントを再起動する

## ラウンド終了時のステータスサマリー

各ラウンドの終了時に以下をユーザーに表示する:

```
**ステータス**
- 現在のフェーズ: X/6（フェーズ名）
- ユースケース数: N
- 概念オブジェクト数: N
- 次のアクション: [具体的な次のステップ]
```

## 途中から再開する場合

ユーザーから途中再開の指示を受けたら `${CLAUDE_SKILL_DIR}/references/context-clear-recovery.md` を参照し、出力ファイルの現在の状態を把握してから作業を再開する。
