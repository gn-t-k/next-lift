# フェーズ7: ドキュメント分割

## 概要

Phase 1〜6で作成したワークスペースファイルを、実装時の参照単位で分割するフェーズ。設計プロセスの記録（ボトムアップ/トップダウン設計メモ、パーキングロット等）と実装に必要な成果物を分離し、成果物のみを独立したファイルに抽出する。

## インプット

- **必須**: Phase 6（最終レビュー）が完了したワークスペースファイル
- **必須**: 分割先ディレクトリのパス（メインエージェントがユーザーに確認済み）

## アウトプット

分割先ディレクトリに以下の5ファイルを生成する:

| ファイル | 内容 |
|---------|------|
| `index.md` | ステータス、要件サマリー、ファイル一覧、実装向けNote |
| `design-principles.md` | コンセプト定義、メンタルモデル |
| `domain-model.md` | ユースケース一覧、タスク表、コンテンツ構造 |
| `views-and-navigation.md` | フレーム構造、ナビゲーション構造 |
| `design-decisions.md` | 設計判断ログ |

加えて、ワークスペースファイルの先頭にアーカイブ注記を追加し、ステータスを7/7に更新する。

## 前提知識

### 成果物とプロセス記録の区別

ワークスペースファイルには2種類のコンテンツが混在している:

- **成果物**: 実装時に参照が必要な設計の結論。分割先ファイルに含める
- **プロセス記録**: 設計過程で有用だったが実装には不要な中間成果物。ワークスペースファイル（アーカイブ）にのみ残す

### コンテンツマッピング

#### `index.md` に含めるもの

| ワークスペースのセクション | 説明 |
|--------------------------|------|
| ステータス | 7/7に更新した状態 |
| 要件サマリー | システム名、プラットフォーム、入力資料等 |
| Phase 6 レビュー結果のNote | 実装時に確定が必要な事項（NOTE-1〜） |

加えて、分割先ファイルの一覧と各ファイルの説明を記載する。

#### `design-principles.md` に含めるもの

| ワークスペースのセクション | 説明 |
|--------------------------|------|
| Phase 4 > コンセプト定義 | ユーザー体験、設計思想、提供価値、コンセプトイメージの全体 |
| Phase 4 > メンタルモデル | 全グループ、最重要の選定理由、補足 |

#### `domain-model.md` に含めるもの

| ワークスペースのセクション | 説明 |
|--------------------------|------|
| Phase 1 > ユースケース一覧 | 全グループ（A〜E等） |
| Phase 2 > タスク表 | Feature一覧と全Function表 |
| Phase 3 > コンテンツ構造 | 概念オブジェクト一覧、パラメータパターン、命名規則、導出ビュー、コンテンツ構造図（Mermaid）、関連の補足 |

#### `views-and-navigation.md` に含めるもの

| ワークスペースのセクション | 説明 |
|--------------------------|------|
| Phase 4 > フレーム構造 | ビュー表現、恒常領域、グローバルナビゲーション方針、レイアウト基本方針、単位ビュー一覧、アクション配置 |
| Phase 5 > ナビゲーション構造 | 多重度からのベースパターン導出、挟み込み統合結果、兄弟間ナビゲーション、横断到達の視覚的優先度、ナビゲーション構造図（iOS/Web別Mermaid）、ビュー到達経路サマリー |

#### `design-decisions.md` に含めるもの

| ワークスペースのセクション | 説明 |
|--------------------------|------|
| 設計判断ログ | 全件（取り消し線付きの判断も含む） |

#### 分割先に含めないもの（プロセス記録）

以下はワークスペースファイル（アーカイブ）にのみ残る:

| ワークスペースのセクション | 除外理由 |
|--------------------------|---------|
| Phase 5 > ボトムアップ設計メモ | 統合結果に集約済み |
| Phase 5 > トップダウン設計メモ | 統合結果に集約済み |
| Phase 3・4 > パーキングロット対応 | 決定内容は設計判断ログに記録済み |
| Phase 6 > レビュー結果（Critical/Improvement） | 修正適用済み。適用内容は設計判断ログに反映済み |
| Phase 6 > 検証チェックリスト | 検証完了済みのプロセス記録 |
| パーキングロット | 全件解決済み |
| Phase 3 > 除外した候補 | 設計過程の記録。判断は設計判断ログに反映済み |

### 参照単位の設計意図

各ファイルは以下の利用シーンを想定している:

- **`design-principles.md`**: 実装を始めるとき最初に読む。設計の「なぜ」と「どういう体験を目指すか」を理解する
- **`domain-model.md`**: ドメインオブジェクトの構造を確認するとき読む。「何が存在し、どう関連するか」を理解する
- **`views-and-navigation.md`**: 特定のビューを実装するとき読む。「何をどこに置き、どう遷移するか」を理解する
- **`design-decisions.md`**: 「なぜこの設計になっているのか」を調べるとき読む。オンデマンドで参照する

## 作業手順

1. **ワークスペースファイルを全文読み込む**: ステータスセクションでPhase 6が完了していることを確認する。完了していない場合は中断してメインエージェントに報告する

2. **分割先ディレクトリを作成する**

3. **`index.md` を作成する**:
   - ステータスを7/7に更新して記載
   - 要件サマリーを転記
   - 分割先ファイルの一覧と各ファイルの説明を記載
   - Phase 6レビュー結果のNoteを「実装時の確認事項」として転記

4. **`design-principles.md` を作成する**:
   - Phase 4のコンセプト定義セクションを転記
   - Phase 4のメンタルモデルセクションを転記
   - 見出しレベルを調整（Phase内のH3→ファイル内のH2等）

5. **`domain-model.md` を作成する**:
   - Phase 1のユースケース一覧を転記
   - Phase 2のタスク表（Feature一覧 + 全Function表）を転記
   - Phase 3のコンテンツ構造を転記（概念オブジェクト一覧、パラメータパターン、命名規則、導出ビュー、構造図、関連補足）
   - パーキングロット対応セクションと除外した候補は含めない

6. **`views-and-navigation.md` を作成する**:
   - Phase 4のフレーム構造セクションを転記
   - Phase 5のナビゲーション構造を転記（多重度ベースパターン、挟み込み統合結果、兄弟ナビ、横断到達優先度、構造図、到達経路サマリー）
   - ボトムアップ設計メモ、トップダウン設計メモは含めない
   - パーキングロット対応セクションは含めない

7. **`design-decisions.md` を作成する**:
   - 設計判断ログテーブルを全件転記（取り消し線付きも含む）

8. **ワークスペースファイルを更新する**:
   - ファイル先頭にアーカイブ注記を追加: `> **アーカイブ**: このファイルは設計プロセスの記録です。実装時の参照には `{分割先ディレクトリ}/index.md` を使用してください。`
   - ステータスの「現在のフェーズ」を `7/7（ドキュメント分割完了）` に更新
   - ステータスの「次のアクション」を `実装フェーズへ（参照: {分割先ディレクトリ}/index.md）` に更新
   - Phase 7セクションの生成ファイルチェックリストを全て完了に更新

9. **セルフレビュー**: 以下を検証する
   - コンテンツマッピングに記載された全セクションが、いずれかの分割先ファイルに含まれていること
   - プロセス記録が分割先ファイルに混入していないこと
   - Mermaid図のシンタックスが壊れていないこと
   - ファイル間の参照（設計判断番号等）が正しいこと

## 戻りフロー

- **条件**: ワークスペースファイルのステータスがPhase 6完了でない場合
- **手順**: 作業を中断し、メインエージェントに報告する。Phase 6を先に完了させる
- このフェーズは機械的な抽出作業のため、設計判断への戻りフローは発生しない

## アンチパターン

- **コンテンツの要約・書き換え**: 抽出時にコンテンツを要約したり、表現を変えたりする。見出しレベルの調整とファイル間参照の追加のみ許可。成果物の内容はそのまま転記する
- **プロセス記録の混入**: ボトムアップ/トップダウン設計メモ、パーキングロット対応、解決済みレビュー指摘、検証チェックリストを分割先ファイルに含める
- **設計判断の取捨選択**: 取り消し線付きの判断を省略する。取り消しの経緯自体がトレーサビリティ上の価値を持つ
- **ファイル間の依存関係**: 分割先ファイルの内容を理解するために別の分割先ファイルの読み込みが必須になる構造を作る（`design-decisions.md` への参照は例外）
