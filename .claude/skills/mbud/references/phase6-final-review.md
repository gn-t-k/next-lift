# フェーズ6: 最終レビュー

## 概要

全フェーズの成果物を横断的に検証し、品質を確保するフェーズ。全フェーズの評価基準を統合した包括的チェックリストで検証する。

## インプット

- **必須**: 全フェーズの成果物（ユースケース一覧、タスク表、コンテンツ構造、コンセプト定義、メンタルモデル、フレーム構造、ナビゲーション構造）

## アウトプット

- **レビューレポート**（Critical / Improvement / Note の3段階で分類）
- 出力ファイルの「Phase 6: レビュー結果」セクションに記録

## 前提知識

### 指摘事項の分類

- **Critical**: 成果物間の矛盾、モードレス原則違反、ユースケース根拠の欠如など、設計の根幹に関わる問題
- **Improvement**: 表現の改善、パターン活用の余地、命名の洗練など、品質向上の提案
- **Note**: 今後の検討事項、プラットフォーム固有の考慮点、将来の拡張に関するメモ

## 作業手順

1. **成果物間の整合性を検証する**
2. **モードレス原則を検証する**
3. **プラットフォーム適合性を検証する**
4. **拡張性を検証する**
5. **指摘事項を分類・記録する**

## 検証項目

### 1. 成果物間の整合性

- [ ] すべてのタスクがユースケースに対応しているか
- [ ] すべての概念オブジェクトがユースケースまたはタスクに根拠があるか
- [ ] ナビゲーション経路がコンテンツ構造の多重度と整合しているか（nを含む関連に一覧があるか、0を含む関連に空表示があるか）
- [ ] フレーム構造がコンセプト定義に沿っているか
- [ ] メンタルモデルの最重要項目（★）がファーストビューに反映されているか
- [ ] 設計判断ログに根拠のない判断がないか

### 2. モードレス原則

- [ ] ファーストビューにオブジェクト（名詞）が並んでいるか（動詞が並んでいないか）
- [ ] タブ/メニュー名が概念オブジェクトの名詞か（「〜管理」「〜機能」でないか）
- [ ] 操作手順が「名詞→動詞」になっているか（「動詞→名詞」のモーダル方式でないか）
- [ ] インターフェイスをブロックしていないか（フルスクリーン、アプリケーションモーダル）
- [ ] 多重モーダルが2〜3階層以内か
- [ ] モーダルダイアログにバツボタンがないか（「キャンセル」と「肯定アクション」の2択か）
- [ ] 通知がインライン（バナー等）で控えめか（モーダルダイアログで通知していないか）
- [ ] モーダルなコマンド名の末尾に「…」が付いているか（別タスクがあることを示唆）
- [ ] ユーザーがモードから脱出できる「避難口」（Escape、キャンセル等）が明確か

### 3. プラットフォーム適合性

- [ ] レイアウトの情報設計原則に沿っているか（書字方向に従い、主=粗、従=細）
- [ ] プラットフォーム固有のイディオムを活用しているか
- [ ] 新しい操作が1回で覚えられるイディオムになっているか
- [ ] フォームファクタに適したレイアウト方式（アダプティブ/レスポンシブ）を選択しているか
- [ ] iOSとAndroidのデザインを無理に統一しようとしていないか
- [ ] フローティングツールバーが情報を隠していないか
- [ ] コンテクストメニュー主体の設計になっていないか
- [ ] ドラッグ＆ドロップや長押しが必須操作になっていないか（代替手段があるか）

### 4. 拡張性

- [ ] 「1つ」とされているものが本当に1つか疑っているか
- [ ] 空の状態、並び順変更、後からの追加を想定できているか
- [ ] 寿命の長い基盤と寿命の短い表層を区別しているか
- [ ] パターンが再利用されているか（同じ意味に同じ表現）

## 戻りフロー

- **条件**: Criticalレベルの指摘が発見された場合
- **手順**: 該当フェーズのサブエージェントを再起動して修正する → 修正後に最終レビューを再実行する
- **記録**: 指摘内容、修正内容、修正理由を設計判断ログに記録する

## アンチパターン

- **見た目の評価に終始する**: 構造設計の品質（成果物間の整合性、モードレス原則）を評価せず、ビジュアルの好みだけで判断する
- **チェックリストの機械的な消化**: 各項目の意味を考えずにチェックを付ける
- **Criticalの放置**: 根幹に関わる問題を「後で直す」として先送りする
