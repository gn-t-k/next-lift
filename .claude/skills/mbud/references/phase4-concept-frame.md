# フェーズ4: コンセプト定義・フレーム構造設計

## 概要

デザインの「芯」（コンセプト）を定め、ユーザーのメンタルモデルを整理し、UIの「外枠」（フレーム構造）を設計するフェーズ。

3つのパート（A: コンセプト定義、B: メンタルモデル整理、C: フレーム構造設計）から成る。

## インプット

- **必須**: ユースケース一覧（Phase 1）、タスク表（Phase 2）、コンテンツ構造（Phase 3）
- **参考**: ビジネス要求、行動シナリオ

## アウトプット

- **コンセプト定義**（テキスト）
- **メンタルモデル**（アウトライン形式）
- **フレーム構造**（テキストによる構造記述）
- 出力ファイルの「Phase 4: コンセプト定義・フレーム構造」セクションに記録

## 前提知識

### コンセプトとは

デザインの「芯」となる指針。トレードオフが生じた際の判断基準となり、チーム全体で目指すべきゴールを共有するためのもの。

### メンタルモデルとは

ユーザーが心の中で思い描く暗黙のイメージ。「何を見て、何ができるのか」の構造。UIの構造がメンタルモデルと一致していると「直感的で使いやすい」と感じる。

### フレーム構造とは

コンテンツ構造を取り囲むUIの「枠組み」。コンテンツや具体的なコントロール要素を取り除いた簡素なワイヤーフレームとして表現する。ビジュアルデザイン（色、タイポグラフィ等）は含めない。

### 単位ビューとは

ユーザーがメンタルモデルとして「意味のあるビューのまとまり」として認識できる範囲。画面全体だけでなく、リストのカード、スクロールビュー、モーダルビューなども1つの単位ビューとして扱う。

## 作業手順

### Part A: コンセプト定義

1. **提供したいユーザー体験を定義する**: 「〜になって欲しい」と丸め込んだ一言で記述する。「〜させる」とユーザーを縛る表現は使わない
2. **設計思想を明示する**: トレードオフが生じた際の判断基準を記述する（例: 「ユーザビリティを多少犠牲にしてでもセキュリティを厳しくする」）
3. **製品の提供価値を列挙する**: 何をするものなのかを端的な言葉で箇条書きにする（例: 「予定を立てる」「メッセージを受信する」）
4. **コンセプトイメージを言語化する**: デザインの姿やインタラクションの様子を記述する

### Part B: メンタルモデル整理

1. **リスト化する**: 概念オブジェクト、タスク、コンセプト（提供価値）の3つのリストを並べる
2. **親子関係を構築する**: コンセプトを親にして、関連するタスクをぶら下げるツリー構造を作る
3. **依存関係を明示する**: タスクと概念オブジェクトを紐づけ、「どのタスクがどのオブジェクトに作用するか」を隅付き括弧【 】で併記する
4. **最重要項目を決定する**: 全体で最も重要なコンセプト/タスク/オブジェクトに★マークを付ける。最重要は**1つに絞る**

出力フォーマット例:
```
★メッセージを受信する       【メッセージ】
　　受信
　　アカウント切り替え

メッセージの下書きを編集する 【メッセージ】
　　新規作成
　　編集
　　下書き保存
```

### Part C: フレーム構造設計

1. **コンテンツ構造からビュー表現を検討する**: 概念オブジェクトの特徴から適したビューを考える（大量→リスト、動画→プレイヤー等）
2. **ユースケース/コンセプトからビュー表現を検討する**: 重視するユースケースの概念オブジェクトを中心に露出させる。ユーザーが能動的か受動的かで検索UIの出し方を検討する
3. **恒常領域を列挙する**: どのビューでも常に表示される領域（グローバルナビゲーション、通知領域、ステータスバー等）を定義する
4. **グローバルナビゲーションの方針を決める**: アプリ全体を貫くナビゲーション構造を**1つに絞る**
5. **レイアウトの基本方針を定める**: マルチウインドウ、マルチペイン（サイドバー、コンテンツエリア等）の基本ルールを定める
6. **単位ビューを決定する**: ユースケースやタスクを1つ実現するための意味あるビューのまとまりを定義する
7. **アクション配置を対応付ける**: 各単位ビューにCRUDに基づくアクション（追加、削除等）をどこにどう配置するか紐づける

## 戻りフロー

- **条件**: コンセプト定義がユースケースの優先度と矛盾する場合
- **手順**: ユースケースの優先度を再評価するか、コンセプトを調整する

- **条件**: フレーム構造検討中に概念オブジェクトの不足を発見した場合
- **手順**: Phase 3に戻りコンテンツ構造を拡充する

- **記録**: 変更内容と理由を設計判断ログに記録する

## アンチパターン

- **コンセプト「〜させる」**: ユーザーを縛る表現。「〜になって欲しい」を使う
- **最重要コンセプトの複数設定**: 優先度が判断できなくなる
- **グローバルナビゲーション構造の複数設定**: 同じアプリ内で異なるナビ構造を持つとユーザーがルール理解に苦痛を覚える
- **詳細なワイヤーフレーム**: 色、タイポグラフィ、具体的なUIコントロールの詳細をフレーム構造に含めない
- **タスク起点のUI**: ファーストビューに「追加する」「管理する」を並べない。オブジェクト（名詞）を置く
