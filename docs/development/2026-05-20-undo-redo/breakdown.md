# Breakdown: undo/redo（取り消し可能性の担保）

## ステータス

- 状態: スタブ（要件未確定）
- 現在のフェーズ: 0/5
- 最終更新: 2026-05-20

## 要件サマリー

- 元となるプロダクト方針: `docs/project/ui-design/design-principles.md`
  - 「**安心の設計**: undo/redo、自動保存で『壊れない』。取り返しのつかない操作は結果を明示」
  - 「**発見の設計**: ... 失敗の心理的障壁を下げておく（undo、安心の設計）」
- 関連する設計判断:
  - ui-design 設計判断 #71（2026-05-20 更新）: 種目計画削除を直接ボタンに統一。誤タップ時のリスクは本 breakdown の undo/redo で担保する前提
  - ui-design 設計判断 #34: 種目削除は完全削除 + 警告ダイアログ。undo/redo があれば警告のみで取り消し可能性が成立する
  - ui-design 設計判断 #47: 1RM 削除時の %1RM セット計画への影響保持（可逆性の重視）
- スコープ（暫定）:
  - 削除操作の取り消し（種目計画 / セット計画 / Day / プログラム / 種目 / 1RM / ワークアウト / セット記録）
  - 必要なら値編集の取り消し（インライン編集等）も対象に含めるか後で判断
- スコープ外（暫定）:
  - 認証・サインアウト等のシステム操作

## 背景

V2 ProgramDetail で種目計画の削除 UI を Menu → 直接ボタンに統一した（設計判断 #71、2026-05-20）。直接ボタン化で操作コストは下がる一方、親階層（子セット計画を持つ）の削除は影響範囲が大きいため、誤タップ時の取り消し手段が必要。

design-principles.md には「安心の設計: undo/redo」が明示されているが、実装タスクとしてはまだ立っていない。本 breakdown で取り消し可能性の実装方針を整理する。

## 未確定事項

要件詰めのときに決める論点を列挙:

### 仕様面

- どの操作を undo/redo の対象にするか
  - 削除のみ？ 値編集も？ 順序変更も？
  - インライン編集（プログラム名 / Day ラベル / セット計画値）は draft + Escape で既に取り消し可能。これと undo/redo の関係をどう整理するか
- スコープの単位
  - グローバル（アプリ全体で1本のスタック）
  - ビュー単位（V2 では V2 内、V6 では V6 内）
  - 操作対象オブジェクト単位
- 永続性
  - メモリのみ（リロード or ページ遷移で消える）
  - 永続化（ローカル / サーバー）
- 履歴の深さ（何手戻れるか）
- セッション跨ぎでも undo できるか
- 他デバイスからの変更との競合（Per-User DB で複数端末同時編集が想定される）

### UI 面

- 取り消し導線
  - Toast「取り消す」ボタン（Gmail / Material 風）
  - キーボードショートカット（Cmd/Ctrl+Z）
  - グローバルな履歴パネル
- iOS とWeb で UI が違ってよいか（iOS はシェイク to undo 等）

### 実装面

- DB レベルの取り消し戦略
  - 物理削除 + history テーブル
  - 論理削除（soft delete）+ 復元
  - イベントソーシング
- Per-User DB の制約とどう折り合うか
- apps/web と apps/ios で実装をどこまで共有するか
