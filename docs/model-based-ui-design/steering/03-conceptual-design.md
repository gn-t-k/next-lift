# 概念設計

モデルベースUIデザインにおける「概念設計」は、UIの構造設計プロセスの中盤に位置し、扱う情報コンテンツや機能の概要を明確にする重要なフェーズです。この段階では、UIの「中身」と「枠組み」という2つの軸で設計を進め、同時にUIのコンセプトも定義することで、UIデザイン全体の強固な基盤を築くことを目指します。

「概念設計」を構成する主な要素は以下の3つです：

1. **UIの中身の定義（コンテンツ構造）**：情報コンテンツのモデルである「コンテンツ構造」を構築します。
2. **UIの枠組みの定義（フレーム構造）**：コンテンツ構造を取り囲む「フレーム構造」を構築します。
3. **デザインの芯の定義（コンセプト）**：デザインの指針を示す「コンセプト定義」を行います。

## 概念設計の意義

概念設計は、単に設計を効率化するだけでなく、ユーザー体験を考慮したデザインの達成や、エンジニアリングチームとの共通言語の醸成にも貢献します。

- **設計効率の向上**：モデルを用いて検討するため、迅速かつ効率的に設計を進められ、設計変更も容易になります。
- **ユーザー体験との関連付け**：ユーザーの行動シナリオからキーワードを抽出し、概念オブジェクトを定義することで、ユーザー体験に基づいた設計を実現します。
- **エンジニアリングとの連携強化**：「概念モデル」を共通言語として用いることで、デザインとエンジニアリングの間での概念認識や命名の統一が図られ、UI上の表現とプログラム内の表現を一致させやすくなります。

## コンテンツ構造の設計

コンテンツ構造は、サービスやアプリ内で扱うコンテンツに関する構造設計とその成果物（概念モデル）です。これはUIの「中身」を決定する過程であり、ユーザーが直接触れるUIの形に直結するものではありませんが、UIのコンセプトを形成する上で非常に重要な工程となります。

### 概念オブジェクトの抽出と定義

概念オブジェクトは、ミュージックアプリにおける「ミュージックトラック」や「アルバム」のように、名詞で表せる概念単位です。これらは、事前に整理された「行動シナリオ」や「ユースケース」、そして「タスク」といった情報から「抽出」できます。とくに、ユースケースを「〈誰〉が〈何〉を〈行動〉できる」という構文で表現した場合、この「〈何〉」に当たる目的語が概念オブジェクトの候補となります。

概念オブジェクトの要件としては、以下の点が挙げられます：

- 名詞で表せる「もの」であること。
- コマンド（アクション）やジェスチャーそのものではないこと。
- 認知や操作の対象となり得ること。
- メタファーで表されることがあること。
- メタデータを含む可能性があること。
- プリミティブな値ではないこと。
- CRUD（作成、読み込み、更新、削除）の対象であること。
- 複製できる可能性があること。
- 同様の性質を有したものが複数個存在し得ること。

### 概念オブジェクトへの命名

概念オブジェクトに名前を付ける際は、実装内部の表現や専門用語を避け、ユーザーが直接認知し、理解しやすい表現にすることが原則です。アプリが象徴するメタファーやコンセプトを念頭に置くことで、ユーザーにとって馴染みやすいUI表現に近づけられます。たとえば、不要なファイルを溜めておく場所を「ゴミ箱」と表現することは、ユーザーの日常の経験に基づいたメタファーの活用です。また、将来的な拡張性を考慮し、意味が狭すぎる名前を避けることも重要です。

### 用語の定義（シソーラス定義）

概念オブジェクトや機能など、デザインに関わるさまざまな概念の名前や語彙は、「文言定義表」にまとめて管理することが推奨されます。これはデザインとエンジニアリング双方での語彙の統一を目指すもので、「ユビキタス言語」の醸成にも繋がります。

### キーオブジェクトの特定

複数の概念オブジェクトが定義されたら、その中でもっとも重要または主要と考えられるものを「キーオブジェクト」（または「メインオブジェクト」）として特定します。これは、ユースケースやタスクの出現頻度、もしくはアプリのコンセプトにおいてもっとも重視したい存在であるかどうかに基づいて判断します。たとえば、ミュージックプレイヤーであれば「ミュージックトラック」や「プレイリスト」、料理アプリであれば「レシピ」などがキーオブジェクトになり得ます。

### 概念オブジェクト間の関連性の定義

コンテンツ構造のモデルを定義するには、抽出した概念オブジェクト同士の関連性（関係）を明確にする必要があります。これは、オブジェクトAから見たオブジェクトBがどのような関係にあるかを「属性名」として示し、矢印で結ぶことで表現します。相互に関連がある場合は双方向の矢印で示すことができます。

### 多重度の明示

関連性を示す際には、「多重度」も併記します。多重度とは、ある2つの概念オブジェクト間の関連において、矢印の先の概念オブジェクトがいくつ存在し得るかを示す情報です。モデルベースUIデザインでは、以下の4パターンで多重度を表します。

- `0..1`：0個または1個。
- `1`：1個のみ。
- `0..n`：0個以上、単数または複数。
- `1..n`：1個以上、単数または複数。

多重度の情報からは、UIの表示パターンをある程度予測できます。

- 「0」が含まれる場合：空表示を検討する必要があります。
- 「n」が含まれる場合：一覧または複数個並べた表示を検討する必要があります。
- 最大値が「1」の場合：単体表示を検討する必要があります。

このルールを覚えておくことで、UIの絵を描く前に、概要一覧・詳細表示・空表示といった大まかな表示パターンを漏れなく見積もることが可能になります。

### プロパティの記述

コンテンツ構造の概要レベルの記述だけでは情報が不足する場合、概念オブジェクトが持つ詳細な情報（メタデータなど）を「プロパティ」として追記することで、さらに情報を拡充できます。

## フレーム構造の設計

フレーム構造は、UIの基本的な枠組みを決定するモデルです。コンテンツ構造がUIの「中身」を定義するのに対し、フレーム構造はそれを包むUIの「外枠」に焦点を当てます。これはナビゲーション構造の材料にもなり、UIの骨格を構築する上で重要です。

### ワイヤーフレームの活用

フレーム構造の成果物は主に「ワイヤーフレーム」として表現されますが、モデルベースUIデザインでは、具体的なUI部品やレイアウト、色、タイポグラフィ、詳細なインタラクション要素などをワイヤーフレームの設計から基本的に除外します。その代わりに、コンテンツ構造をどのように表示する枠組みや、グローバルナビゲーションの方針（タブバーやナビゲーションメニューの扱いなど）を検討します。表現度合いは簡素で十分とされています。

### 単位ビューの定義

「単位ビュー」は、ユーザーがメンタルモデルとして「意味のあるビューのまとまり」と認識できる範囲を指します。これはUIの最小構成単位であり、リストやグリッドのカード単位、ヘッダー、ナビゲーションバー、モーダルビューの単位などが含まれます。単位ビューは具体的なユースケースやタスクを実現するための意味を持ちます。

### ビュー表現の検討

コンテンツ構造やユースケース、コンセプトから、UIに適切だと考えられるフレーム表現を導き出します。たとえば、大量の情報を扱う概念オブジェクトにはリストやグリッド、スクロールビューを組み合わせる、といった検討を行います。ユーザーが情報に「受動的」に接するのか、「能動的」に接するのかによっても、適切なビュー表現は異なります。

### グローバルナビゲーションの方針

コンテンツ構造の設計を終えると、情報アーキテクチャの形がある程度見えてくるため、ナビゲーションの戦略も検討しやすくなります。グローバルナビゲーションの構造はアプリのUIに「芯」を通す屋台骨となるため、アプリ全体で機能する「1つ」のグローバルナビゲーション構造を設けることが重要であり、後の変更が起きないように慎重に決定します。

### レイアウトの基本方針

ここでいうレイアウトは、個々の要素の整列や行間といった細かいものではなく、ビューを区切る「ペイン」の基本ルールを決めることを指します。とくにデスクトップUIでは、グローバルナビゲーション用、ツールバー用、コンテンツ描画用、プロパティ操作用といったペインの役割を決めておく必要があります。

## コンセプト定義

コンセプト定義は、デザインの「芯」となり、デザイナーがどのようなイメージを形にしていくのか、その指針を示します。コンセプトがしっかり定まっていれば、デザインに大きなブレが生じにくくなります。

コンセプト定義は、以下の4つの柱で構成されます。

1. **提供したいユーザー体験の明示**：ユーザーに「どういう体験をして欲しいのか」を一言で書き留めます。ユーザーの思考や行動を断定的に表現するのではなく、「〜になって欲しい」といった可能性の含みを持たせた表現を用いることが重要です。
2. **設計思想の明示**：デザインにおいてもっとも重視する考え方を明確にします。これは、トレードオフが生じた際に意思決定の指針となるものです。たとえば、「セキュリティを最優先し、ユーザビリティを一部犠牲にする」といった考え方です。
3. **何をするものなのかの明示**：製品が何を提供しようとしているのか、またはユーザーが何をしようとしているのかを表す大きなコンセプトを示します。これはタスクに近いですが、より端的にできることを表現します。
4. **コンセプトイメージの描画**：デザイナーが思い描く製品やUIの姿を、落書きレベルの簡単な絵で表現します。これは、具体的な設計を検討したり、開発チーム内での議論に活用したりする上で非常に有効です。

定義されたコンセプトは、デザインプロセス全体を通して繰り返し参照され、現在の設計がコンセプトから逸脱していないかを確認するために用いられます。これにより、一貫性があり、堅牢なデザインを実現できます。

## 他の設計フェーズとの連携

概念設計は、モデルベースUIデザインの他のフェーズと密接に連携しています。

- **序盤フェーズ**：ユーザーリサーチの結果から得られた「ユースケース定義」や「タスク整理」の成果物（行動シナリオやペルソナモデル、タスク表など）を、概念オブジェクトやタスク、コンセプトの抽出・整理の材料として活用します。
- **中盤フェーズ**：概念設計で確立されたコンテンツ構造やフレーム構造、コンセプトは、その後の「ナビゲーション構造設計」の基盤となります。とくに、概念オブジェクトの多重度がナビゲーションパターン（一覧表示、詳細表示、空表示）の検討に直接影響します。
- **終盤フェーズ**：構築された概念モデルは、「プラットフォームへの適合とインタラクション設計」において、具体的なレイアウトやインタラクションパターンの検討に活用されます。
