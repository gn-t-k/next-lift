# ADR-024: プリミティブ層ポリシーと2層 + 昇格ルール

## ステータス

Accepted

## 関連ADR

- [ADR-014: Intent UIの採用](./014-intent-ui.md) - 本ADRで運用方針を具体化
- [ADR-015: Tailwind CSS v4の採用](./015-tailwind-css-v4.md)
- [ADR-023: コンテキスト依存UIスタイリング戦略](./023-context-dependent-ui-styling.md)

## コンテキスト

ADR-014 で Intent UI（copy-and-own モデル、React Aria Components ベース）を採用した。その後 `packages/react-components/` に Button / Dialog / Modal 等を取り込んで運用を始めたところ、以下の課題が顕在化した。

### Intent UI コンポーネントに「意見」が混入している

実例として `Modal` を取り込んだ際、画面幅によって **dialog と drawer を自動的に切り替える** 挙動が含まれていた。しかしダイアログかドロワーかの選択はアプリケーションのユースケース・画面構成によって決まるべきもので、**UIコンポーネントライブラリが一律に決めてよい意見ではない**。結果として、コピー後に大幅な書き換えを行ってこの挙動を剥がした。

この経験から、以下の気付きを得た:

- Intent UI のコンポーネントは「プリミティブ（意見を持たないアクセシビリティ + 基本挙動）」と呼べる粒度ではないものが混ざっている
- 真のプリミティブ基盤は Intent UI ではなく、その下で使われている **React Aria Components**
- Intent UI は「React Aria + Tailwind の使い方を示すテンプレート集」として扱うのが実態に合う

### ディレクトリ命名 `src/ui/` の問題

現行の `packages/react-components/src/ui/` という命名は、あらゆる階層のコンポーネント（プリミティブ、ドメインコンポーネント、ビュー）が等しく「UI」であるため、**最下層を指す語として機能していない**。命名が配置基準を導けない。

### ビュー層の導入に伴う階層設計

`docs/project/ui-design/` で定義された 14 個の単位ビュー（V1〜V15、V7廃止）を `packages/react-components/` に実装するにあたり、プリミティブ層の上にどのような階層を置くかを決める必要があった。3層（プリミティブ / 中間 / ビュー）構造を検討したが、中間層を `components/` と命名すると `ui/` と同じ「あらゆるコンポーネントに当てはまる語」という問題を抱えることが判明した。

## 決定内容

以下の4点を決定した。

### (a) プリミティブ層の真のベースは React Aria Components

Intent UI はプリミティブライブラリではなく、**起点テンプレート集** として扱う。

- `pnpm ui:add` で Intent UI から取得後、意見が混入していれば剥がす
- 意見が剥がしきれない粒度のものは React Aria Components から直接書き直す
- どちらの出自であっても、`src/primitive/` に置かれる時点で「自分たちのコード」

プリミティブ層の定義: **意見を持たない、アクセシビリティと基本挙動のみを提供するコンポーネント**。ドメイン用語を名前に含まない。

### (b) `src/ui/` → `src/primitive/` リネーム

`ui` はあらゆる階層に当てはまる語で、最下層を指す語として機能していない。プリミティブ層を指す語として `primitive` を採用する。

### (c) 2層構造（`src/primitive/` + `src/view/`）+ コロケーション

中間層（いわゆるドメインコンポーネント層）を独立したディレクトリとしては設けない。代わりに、**特定ビュー専用のサブコンポーネントは最初に使うビューのディレクトリ配下にコロケーション** する。

```text
packages/react-components/src/
├── primitive/
│   ├── button.tsx
│   ├── text-field.tsx
│   └── ...
└── view/
    ├── program-list/
    │   ├── index.tsx              # V1 プログラム一覧
    │   └── program-list-item.tsx  # V1 専用のサブコンポーネント
    ├── program-detail/
    │   └── index.tsx              # V2
    └── ...
```

基本の配置基準:

1. V1〜V15 のどれかに対応する → `src/view/{ビュー名}/index.tsx`
2. 特定ビュー専用のサブコンポーネント → `src/view/{ビュー名}/` 配下にコロケーション
3. ドメイン用語（Program / Day / ExercisePlan / SetPlan / Workout / Exercise / 1RM など）を含まない汎用部品 → `src/primitive/`

### (d) 複数ビューで使われるようになった時点で「昇格」する

コロケーションされたサブコンポーネントが 2 つ目のビューで使われることになった時点で、**そのコンポーネントを昇格させる**。ビュー専用ディレクトリから上位ディレクトリへ引き上げ、所有権を配置で表現する。

昇格先の判定:

- **ドメイン用語を剥がして汎用化できる** → `src/primitive/` に昇格（一般名に改名）
- **ドメイン用語を保ったまま共有される** → `src/view/` 直下に昇格

```text
src/
├── primitive/
│   └── numeric-value.tsx       # ドメイン用語を剥がして昇格した例
└── view/
    ├── program-list/
    │   └── index.tsx
    ├── program-detail/
    │   └── index.tsx
    ├── day-detail/
    │   └── index.tsx
    └── set-plan-row.tsx        # V3, V10 などで使う → view/ 直下に昇格
```

**昇格のタイミング**: 「2件目のビューから import したい」と感じた時点。見かけ上使えそうというだけで先回りしない（YAGNI）。

**理由**:

- **所有権が配置で分かる**: `src/view/{ビュー名}/` 配下はそのビュー専用、`src/view/` 直下は共有資産。依存方向の誤解が生じない
- **初期ビューが削除されても孤児化しない**: 昇格後は元ビューとの結びつきが切れる
- **ビュー間の内部依存を避ける**: 「ビューA → ビューBの内部サブコンポーネント」という依存経路が発生しない

## 結果・影響

### メリット

- **プリミティブ層の境界が明確**: Intent UI のコンポーネント全体を鵜呑みにせず、意見の有無で判定できる。レビュー基準も揃う
- **命名が配置基準を導ける**: `primitive` は「意見を持たない基礎要素」を指す語として機能する。`ui` のように「何にでも当てはまる」問題を解消
- **配置でオーナーシップが分かる**: `view/{ビュー名}/` 配下はそのビュー専用、`view/` 直下は共有、`primitive/` は汎用。ディレクトリ構造が依存関係を物語る
- **YAGNI 原則との両立**: 初回はコロケーションで手早く配置、2件目の利用で昇格する。前倒しの構造化を避けつつ、共有化のタイミングで必ず整理される
- **ADR-014 との整合**: copy-and-own モデルの精神（ロックインを避けて自分たちのコードとして扱う）を具体的な運用レベルに落とし込む

### デメリット

- **プリミティブ判定の主観性**: 「意見を持たない」という基準には解釈の余地があり、レビュー時に議論が発生する可能性がある（例: コンポーネント名を引数に取るのは意見か、スタイルプリセットは意見か）
- **昇格判断のコスト**: 2件目のビューで使うとき、昇格先（primitive か view 直下か）を毎回判定する必要がある。ドメイン用語を剥がせるかの線引きに迷う余地がある
- **昇格に伴うリファクタコスト**: 昇格時にファイル移動と import パスの更新が発生する。複数のビューで同時に参照されている場合は一括で修正する
- **Intent UI の更新追従が進みにくい**: 意見を剥がす書き換えを行った場合、upstream 更新を取り込む難度が上がる（ADR-014 のデメリットと同質だが、本ADRでより積極的に書き換える方針のため増幅する）

## 代替案

### (1) Intent UI のコンポーネントをそのままプリミティブとして扱う

取り込んだコンポーネントに意見が混入していても、なるべくそのまま使う方針。

- **メリット**: 書き換え作業が不要、upstream の更新を取り込みやすい
- **却下理由**: Modal の dialog/drawer 切替のように、UIライブラリが決めるべきでない意見が混入している。プロジェクトのユースケースに合わず、結局ビュー側で上書きが必要になる。「プリミティブ」という名目と実態が乖離する

### (2) Intent UI を使わず React Aria Components から自作のみ

Intent UI のテンプレートは使わず、すべて React Aria Components から直接書く。

- **メリット**: プリミティブの基準がブレない、外部ライブラリの意見に振り回されない
- **却下理由**: スタイリングのスタートラインが毎回ゼロからになる。ADR-014 で採用した「完成品を使ってスタートする」開発効率の利点を失う。個人開発の時間制約に反する

### (3) 3層構造（`primitive` / `components` / `view`）

プリミティブ層とビュー層の間に、ドメインコンポーネント層（`components/`）を明示的なディレクトリとして設ける。

- **メリット**: 層の責務が一目で分かる、再利用可能なドメイン部品の在りかが明確
- **却下理由**: `components` という名前が `ui` と同じ問題を抱える。primitive も view も components であり、配置基準を名前から導けない。中間層を「ドメイン用語を含む + ナビゲーション目的地でない」という **負の条件の組み合わせ** で定義せざるを得ず、振り分けが機械的にならない

### (4) 最初のビュー配下に置いたまま他ビューから直接 import する

2層 + コロケーションの派生として、複数ビューで使うようになっても配置を変えない方針。

- **メリット**: ファイル移動のコストが発生しない、昇格判断が不要
- **却下理由**:
  - 所有権の誤解: `src/view/program-list/program-list-item.tsx` を V2 が import すると「なぜ V1 配下のものを使う？」という違和感が生じる
  - 最初のビューが削除されたときに孤児化する
  - 依存方向が「ビュー → 別ビューの内部」になってしまう

### (5) 専用の `_shared/` ディレクトリへの強制引き上げ

2件目の利用で `src/view/_shared/` へ必ず移動させるルールを設ける。

- **メリット**: 共有資産が隔離されて一目で分かる
- **却下理由**:
  - 「ビュー専用 vs 共有」の区別は `view/` 内の階層（サブディレクトリ配下 or 直下）で既に表現できる。専用ディレクトリは冗長
  - `_` プレフィックスは特殊な運用ルールを暗示するため、シンプルな `view/` 直下配置の方が理解しやすい

### (6) ディレクトリを切らず完全にフラット

`src/primitive/` も `src/view/` も作らず、`src/` 直下に全てのコンポーネントを並べる。

- **メリット**: 最もシンプル、階層を考える必要がない
- **却下理由**: プリミティブとビューで責務が本質的に異なるため、ディレクトリで区別する価値は大きい。規模が大きくなった際（単位ビューだけで14本）、階層なしでは可読性が下がる

## トレードオフ

プリミティブ層ポリシーと2層 + 昇格ルールは以下のトレードオフを伴う:

- ✅ プリミティブ層の境界明確化、命名の整合性、所有権が配置で分かる、YAGNI との両立
- ❌ プリミティブ判定の主観性、昇格判断のコスト、昇格時のリファクタコスト、Intent UI 更新追従の難度増加

ただし、ADR-014 の copy-and-own の精神を実運用レベルに落とし込むこと、および 14 個の単位ビュー実装にあたって配置基準が機械的に決まることの価値が、判断コストや主観性を上回るため、本ポリシーが最適な選択肢である。
