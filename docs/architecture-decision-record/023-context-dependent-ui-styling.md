## コンテキスト

`docs/project/ui-design/design-principles.md` のコンセプト定義では、Next Liftのビューを**計画時と記録時で異なる見た目・挙動**にする方針が明記されている:

- **カラー**: 計画時は低彩度、記録時は高彩度（ジム環境での視認性確保）
- **タイポグラフィ**: 計画時は細く端正、記録時は数字を太く大きく
- **情報密度**: 計画時は高密度・俯瞰、記録時は低密度・1セットにフォーカス
- **インタラクション**: 計画時は即座・アニメーション控えめ、記録時はアニメーション＋ハプティクス豊か

この「コンテキスト依存」を具体的にどう実装するかが未定だった。実装方法の選択次第で、トークン設計、コンポーネントAPI、配布パッケージの設計すべてが変わる。

また、単位ビューの主役は**数値**（重量・回数・RPE・%1RM・e1RM）であり、一般的なデザインシステムの単一タイポスケールだと「記録時は数字を太く大きく」を適切に表現できない。ドメイン特性に合わせたスケール設計が必要だった。

## 決定内容

以下の6点を決定した。

### (a) 2コンテキスト制

コンテキストを「**ジム外（計画・振り返り）**」と「**ジム内（記録）**」の2つに分類する。意味としては「目を凝らせない・短時間で見切る必要がある環境かどうか」で区別する。

設計原則の「計画時/記録時」という抽象概念を、実装者が迷わず判断できる具体的な基準に置き換えた。

### (b) ビュー単位の境界

1ビュー = 1コンテキスト。ビュー内で混在させない。

単位ビューごとに属するコンテキストを割り当てる:

- **ジム外**: V1, V2, V3, V4, V10, V11, V12, V14, V15
- **ジム内**: V5+V8, V6
- **呼び出し元で切替**: V9（記録直後=ジム内、後日閲覧=ジム外）、V13（V6から=ジム内、V12/V3から=ジム外）

呼び出し元で切替になるビュー本体は1つのコンポーネントで実装し、ルーティング層でラッパーの有無を変える。

### (c) 単一軸で束ねる

彩度・タイポ・情報密度・インタラクションの各差分は**一つの束**として切り替わる。コンテキストを宣言すると全軸同時に変化する。

局所上書きは可能だが、原則として使わない。

### (d) CSS変数swapによる実装

`.context-gym` クラスをビューのルートに付けると、配下のCSS変数の値がクラスセレクタで上書きされる仕組み。React側は `<GymContext>` ラッパーコンポーネントを提供。

```css
/* 概念スキーマ（具体値は tailwind-config 側を参照） */
:root {
  --density-label: <compact size>;
  --density-value: <compact size>;
}
.context-gym {
  --density-label: <comfortable size>;
  --density-value: <comfortable size>;
}
```

配下のコンポーネントは `text-density-label` `text-density-value` 等のクラスを使うだけで、コンテキストに応じて自動的に切り替わる。コンポーネント側に `context` propは持たせない。

### (e) 二重タイポグラフィスケール

テキスト用スケール（Major Third、1.25倍）と**数値専用スケール**（num-sm 〜 num-xl）を別系統で持つ。

テキストスケールは計画ビュー（情報密度高め）向けに控えめなジャンプ率で設計し、数値スケールは別系統でメリハリを確保する。両方を単一スケールで両立しようとすると必ず妥協が生じるため分離した。

### (f) 情報密度トークン（Compact / Comfortable）

コンテキストごとにラベル・値・単位・余白・アクション高さを一括で切り替える。ジム外は Compact、ジム内は Comfortable。

---

トークンの具体値とTailwindクラス名の一覧は [`packages/tailwind-config/tailwind.css`](../../packages/tailwind-config/tailwind.css) および [`packages/tailwind-config/CLAUDE.md`](../../packages/tailwind-config/CLAUDE.md) を参照（本ADRで重複管理しない）。

## 結果・影響

### メリット

- **コンポーネント側がコンテキスト非依存**: `text-density-label` のようなクラスを使うだけで、配下のコンポーネントは context prop を持たない。実装者が毎回コンテキストを意識する必要がない
- **トークン体系がSingle Source of Truth**: `packages/tailwind-config/tailwind.css` に全てのコンテキスト差分が集約される。CSS/Reactどちらの側からも参照可能
- **呼び出し元の責務が明確**: 両コンテキスト対応ビュー（V9/V13）は、ビュー本体を2つ書く必要がなく、ラッパーを付けるかどうかだけで切り替わる
- **ドメイン特性への適合**: 二重タイポスケールにより、本文密度を保ちながら数値だけ主役にできる（記録ビューの体験が明確に改善する）
- **一貫性の担保**: 単一軸で束ねているため、「ジム内モード」を宣言すれば全軸が揃って「ジム向けの顔」になる。`1,2,3がわかれば4を予測できる`というコンセプトと整合

### デメリット

- **トークン数の増加**: 密度依存のセマンティックトークン（`--text-density-*`、`--spacing-density-*`）が追加され、単純な `text-xs` / `text-sm` だけで済む構成より学習コストが上がる
- **局所例外のルールが必要**: 原則ビュー単位だが、例外時は局所的に `.context-gym` をかぶせる運用。暗黙のルールになりがち
- **iOS側との共有コスト**: 現時点ではWeb（CSS変数swap）前提の設計。iOS（React Native）でも同じ意味論を実現するには別途実装が必要（StyleSheetでの動的切替など）。ADR-016のdesign-tokens化構想と合わせて将来検討
- **コンテキスト境界の判断責任**: 新しいビューを追加するたびに「どちらのコンテキストに属するか」を決める必要がある。`design-principles.md` に割り当て表を明記することで対処

## 代替案

### (1) React Contextで伝搬

`<GymContext.Provider value={{ isGym: true }}>` を使い、各コンポーネントが `useContext` で受け取る方法。

- **メリット**: 型安全、Reactの標準パターン、動的な条件分岐が書きやすい
- **却下理由**:
  - コンポーネントが全てコンテキストを意識する必要がある（prop drilling の代替にはなるが、消費側の記述が増える）
  - CSSトークンと分断され、Tailwindユーティリティクラスの力を活かせない
  - SSRでのクライアント/サーバー切替に追加の配慮が必要

### (2) コンポーネント単位でvariant propを受け取る

`<Button variant="gym">` のように各コンポーネントが context を prop で受け取る方法。

- **メリット**: 明示的、変更箇所が分かりやすい
- **却下理由**:
  - 全コンポーネントに variant prop を追加するコストが高い
  - ビュー内の全要素に同じ prop を繰り返し渡す必要がある
  - 「ビュー単位で統一」という原則を型で強制できない（ビュー内で異なる variant を混在させられてしまう）

### (3) 軸ごとに独立したクラス

`.saturation-high`、`.density-comfortable`、`.typo-numeric` のように、軸ごとに独立したクラスを持つ方法。

- **メリット**: 柔軟。例外的な組み合わせに対応できる
- **却下理由**:
  - 組み合わせ爆発（2^n）で管理しきれない
  - 「1,2,3がわかれば4を予測できる」という一貫性のコンセプトと矛盾
  - 実用上、全軸が「ジム環境での視認性確保」という単一目的の裏返しであり、独立に動かす必要がない

### (4) 単一タイポスケール

テキストも数値も同じスケールで済ませる方法（Major Third 1本、または Major Second 1本）。

- **メリット**: シンプル、トークン数が少ない
- **却下理由**:
  - Major Second 一本: 本文階層は整然とするが、記録ビューで数字を目立たせられない（スケール上限1.802remでも迫力不足）
  - Major Third 一本: 数字の主役感は出るが、計画ビューで見出しが浮きすぎ情報密度が下がる
  - ドメインの主役が数値である特性と合わない

### (5) 動的コンテキスト検出（時刻・位置）

時刻帯や位置情報から「今ジムにいる」を自動判定してコンテキストを切り替える方法。

- **メリット**: ユーザーが何もしなくてよい
- **却下理由**:
  - 判定の失敗リスク（ジムに入ったタイミングで誤作動するとUIが急変する）
  - プライバシー（位置情報取得の許諾コスト）
  - 「計画的に記録を伸ばす」というコンセプトで、ジム外でも記録確認することは多い
  - 本質的に「ビューの役割」でコンテキストは決まる（V6はいつ見ても記録モード）。動的判定は不要

## トレードオフ

コンテキスト依存UIスタイリング戦略は以下のトレードオフを伴う:

- ✅ コンポーネントのコンテキスト非依存性、単一軸による一貫性、ドメイン特性への適合、トークンのSingle Source of Truth
- ❌ トークン数の増加、局所例外運用の暗黙ルール、iOS側への横展開コスト

ただし、Next Liftのコアコンセプト（「計画的に記録を伸ばす」「ジム環境での視認性」「一貫性のある設計でユーザーの発見を促す」）への適合が、追加のトークン学習コストを上回るため、本戦略が最適な選択肢である。
